# 测试方案写作原则

本文档记录测试方案修订时需要长期遵守的原则，用于避免后续开发或文档调整破坏验收口径。

## 总体原则

1. 测试方案面向专家评审，优先保证自圆其说、可复现、易理解，不暴露不必要的实现细节。
2. 测试业务的输入参数可以固定。固定参数应写入前置条件和测试步骤，用于保证复核口径一致。
3. 任务运行时间由系统工单的开始时间和结束时间字段输入，不采用人工秒表计时。
4. 业务目标成功率不能只展示最终百分比，应展示每个任务的指标结果、历史基准值、达标阈值、是否达标，以及汇总计算过程。
5. 过程性指标类业务应写明有效样本要求，并与当前可复现实现保持一致。例如矩阵计算业务输入批次数固定为 50、有效吞吐采样点不少于 5 个；视频推理有效统计帧不少于 30 帧；单任务调度时间窗口不小于 5 分钟。

## 用例划分

1. 当前正式测试方案保持 4 个测试用例：矩阵计算业务目标成功率、视频AI推理业务目标成功率、意图解析功能测试、意图解析参数提取准确率测试。
2. 基准测试不单独作为新测试用例，应写入对应业务目标成功率用例的前置条件和测试步骤。
3. 当前业务目标成功率正式验收只写矩阵计算和视频AI推理两个业务，不写模型训练业务。
4. 意图解析功能测试的预期结果保持简洁，表述为系统能正常返回生成解析结果即可。

## 专家版表述边界

1. 组网拓扑由总体测试方案统一给出，本子模块不写当前开发测试环境的具体机器名、IP、节点编号或账号信息。
2. 前置条件采用“相关系统模块部署正常运行”“测试文件已准备”等通用表述，避免写过细部署细节。
3. 专家版测试方案不强调外部接口、批处理平台或厂商控制台等实现名称，统一从“智能体/大模型评测”“系统页面操作”“评测结果留档”的角度描述。
4. 意图解析准确率数据集只做概括说明，不在正式方案中展开具体样本格式、模板分类或构造细节，避免专家围绕数据集细节跑偏。
5. 有 UI 操作的测试步骤应配合截图留档，但截图不宜作为测试用例表的独立字段。推荐做法是在“测试步骤”之后紧跟对应真实截图；系统尚未完善时可短期使用截图占位，正式版必须替换为真实截图。
6. 内部数据集构造应覆盖 8 个模态和 4 类路由策略倾向。正式测试方案只写“覆盖多模态业务、路由策略倾向和缺失/错误参数场景”，不展开每个模板和槽位。
7. 路由策略样本应使用自然语言倾向表达：无特殊偏好对应资源保障，更快/高性能对应完成时间优先，空闲/低负载对应负载均衡，省钱/少占资源对应成本优先；不要要求用户直接说出策略枚举值。
8. 路由 DAG JSON 属于开发调试和外部路由对接材料，不作为专家版意图解析功能测试的页面观察点。功能测试只验证系统能返回结构化解析结果、提示缺失参数并完成工单提交。

## 交付留档

1. 每轮正式评测应记录评测编号、时间、样本总数、成功样本数、准确率或成功率。
2. 业务目标成功率留档应包含批量工单列表、单任务详情、路由结果、部署状态、指标结果和汇总计算截图。
3. 意图解析准确率留档应包含评测页面统计卡、进度区域、样本明细、成功/失败样本示例和下载的评测结果文件。
