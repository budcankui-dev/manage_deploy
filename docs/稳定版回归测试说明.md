# 稳定版回归测试说明

本文定义本系统在进入“稳定版候选”前必须通过的自动化回归。目标是减少“修好一个页面、另一个已跑通功能又坏掉”的反复手测成本。

## 1. 测试分层

### 1.1 小修快测

适用场景：只修复一个明确 bug，且改动范围集中。

执行原则：

- 只跑相关 e2e、相关单测和 `npm --prefix frontend run build`。
- 例如业务测评按钮状态问题，至少跑 `business-tasks-hub.spec.js` 和构建。
- 小修快测通过后可以部署给用户手动确认，但不能直接作为稳定版结论。

### 1.2 稳定版全量前端回归

适用场景：准备给验收方演示、合并大改、部署稳定版、迁移真实网络前。

固定入口：

```bash
cd /Users/yanjia/codes/manage_deploy
./scripts/stable_regression.sh
```

默认本地前端地址是 `http://127.0.0.1:5173`，API 代理默认指向 `http://127.0.0.1:8181`。如果检查远端部署：

```bash
FRONTEND_BASE_URL=http://10.112.244.94:8182 \
PLAYWRIGHT_NO_WEBSERVER=1 \
SKIP_BACKEND_TESTS=1 \
./scripts/stable_regression.sh
```

默认管理员账号使用 `admin/admin`。如需使用专用测试管理员，请先确保该账号已经存在并可登录，然后传入：

```bash
E2E_ADMIN_USERNAME=your-admin \
E2E_ADMIN_PASSWORD=your-password \
./scripts/stable_regression.sh
```

报告默认输出到：

```text
reports/stable-regression/YYYYMMDD-HHMMSS/
```

### 1.3 发布前真实端到端

适用场景：稳定版候选已经通过前端全量回归，需要验证真实容器、IPv6、Node Agent、业务镜像和多节点部署。

发布前真实端到端不放进 `test:stable`，避免每次小修误触发大量工单、容器和远端节点操作。发布前再按使用说明和部署文档执行：

- 用户自启动模式：源端/目的端容器手动启动，平台只部署 compute。
- 批量业务测评模式：系统自动部署 source/compute/sink，跑 30 个测评工单。
- IPv6 容器访问：容器 UI 和容器间业务通信均使用数据面 IPv6。

注意：“能跑通”不等于“验收达标”。发布候选必须额外通过达标门禁：

- 意图解析：固定 `datasets/intent_eval/multi_business.jsonl` 360 条样本，准确率不低于 90%。
- 矩阵乘法计算任务：同一测评轮次可评价工单不少于 30 个，成功率不低于 90%，单工单判定为 `effective_gflops >= 节点基线 * 0.8`。
- 视频 AI 推理任务：同一测评轮次可评价工单不少于 30 个，成功率不低于 90%，单工单判定为 `frame_latency_p90_ms <= 节点基线 * 1.5`。
- baseline 缺失或业务 profile 不匹配时，该工单不计入可评价工单，不能用少量 100% 样本替代正式达标结论。

达标门禁脚本：

```bash
cd /Users/yanjia/codes/manage_deploy
PYTHONPATH=. ./backend/venv/bin/python scripts/acceptance_metrics_gate.py \
  --intent-report reports/intent_eval_online.json
```

业务测评轮次达标检查：

```bash
cd /Users/yanjia/codes/manage_deploy
PYTHONPATH=. ./backend/venv/bin/python scripts/acceptance_metrics_gate.py \
  --api-base-url http://127.0.0.1:8181 \
  --benchmark-run-id high_throughput_matmul-YYYYMMDDHHMMSS \
  --task-type high_throughput_matmul
```

如果同一轮次同时检查两类业务，可重复传 `--task-type`。脚本返回 `passed=false` 时，不能称为稳定验收版。

## 2. 稳定版回归覆盖范围

### 2.1 登录和权限

覆盖目标：

- 管理员登录后进入业务工单中心。
- token 过期后显示中文提示并返回登录页。
- 管理员不能停留在用户端对话页面。
- 用户登录后能进入用户端意图对话页面。

主要测试文件：

```text
frontend/tests/e2e/intent-chat.spec.js
frontend/tests/e2e/business-tasks-hub.spec.js
```

### 2.2 用户端意图对话

覆盖目标：

- 矩阵乘法计算任务：自然语言输入、参数解析、参数完整、确认提交、生成工单。
- 视频 AI 推理任务：自然语言输入、参数解析、参数完整、确认提交、生成工单。
- 参数缺失：不允许提交，页面给出中文补全提示。
- 可用节点按钮：只作为辅助按钮，不占据主聊天区域；计算节点不作为源/目的输入。

当前已覆盖的核心验收路径：

- 矩阵乘法计算任务解析并提交。
- 视频 AI 推理任务解析并提交。
- 视频任务缺少帧率时保持草稿未提交。

### 2.3 管理端业务工单中心

覆盖目标：

- 业务工单中心可打开。
- 工单列表、筛选按钮、清理实例说明可见。
- 侧边栏从业务测评切到拓扑节点、系统设置后仍可正常点击。
- 管理端基础页面不出现空白、前端错误覆盖层或明显布局异常。

### 2.4 业务测评页面

覆盖目标：

- 页面可打开，能展示计算节点基线区域、创建测评工单、运行测评、结果证据区域。
- 点击“开始完整测试流程”后，即使切换左侧菜单再回来，也保持“测评运行中”，不能重复点击。
- 创建工单接口未返回、页面刷新数据未完成时，也保持运行会话锁。
- 历史运行会话恢复后不会重复启动同一轮测评。

注意：自动回归不会默认真实启动 30 个远端容器。正式验收前的 30 个业务测评工单由发布前真实端到端流程执行。

### 2.5 意图评测页面

覆盖目标：

- 页面可打开。
- 评测流程、下载原始数据集、下载评测结果、样本标签和系统解析结果对比区域可见。
- 页面不出现明显横向溢出、中文标题看不清、字段竖排等布局问题。

### 2.6 基础页面巡检

`ui:smoke` 会以管理员和普通用户身份访问并截图：

- `/intent-chat`
- `/business-tasks`
- `/benchmark`
- `/nodes`
- `/templates`
- `/settings`
- `/intent-evaluation`

自动检查：

- 页面是否能加载。
- 是否有横向溢出。
- 中文标签是否被挤成竖排。
- 主要元素是否超出视口。

## 4. 通过标准

稳定版候选至少满足：

- 前端构建成功。
- `test:display`、`test:route-recovery`、`test:benchmark-session` 成功。
- `ui:smoke` 成功，并生成截图。
- Playwright 主流程 E2E 成功。
- 后端关键 API 单测成功，或明确说明跳过原因。
- 发布前真实端到端按当前网络环境单独跑通，并通过 `scripts/acceptance_metrics_gate.py` 达标门禁。

## 5. 后续扩展

下一步可以继续增加：

- 节点 CRUD 前端真实操作 e2e。
- 模板 CRUD 前端真实操作 e2e。
- 用户端工单列表删除、取消、清理容器操作 e2e。
- 意图评测页面“运行一次评测并自动刷新最新轮次”的 e2e。
- 发布前真实容器 E2E 的自动截图和报告汇总。

原则：每次新增一个曾经出过问题的主流程，都要加入稳定版回归，避免下次靠人工再次发现。
