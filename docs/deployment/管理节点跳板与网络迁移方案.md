# 管理节点跳板与网络迁移方案

本文固定当前结论：**人的电脑访问实验网络时，以 compute-1 上的 WireGuard 为主方案；SSH 端口转发只作为应急兜底，不再作为常规路径。**

本文只描述接入和迁移方案，不执行项目迁移。

## 1. 固定结论

系统运行链路和人的调试入口必须分开：

| 链路 | 固定口径 | 是否依赖 WireGuard |
| --- | --- | --- |
| Manager -> MySQL | `172.16.0.254:3306` | 否 |
| Manager -> MinIO | `172.16.0.254:9000` | 否 |
| Manager / 节点 -> Registry | `172.16.0.254:5000` | 否 |
| Manager -> Node Agent | `http://<节点172.16管理IP>:8001` | 否 |
| 容器 -> Manager | `MANAGER_PUBLIC_URL=http://172.16.0.254:8181` | 否 |
| 业务容器 -> 业务容器 | `http://[数据面IPv6]:端口` | 否 |
| 人的电脑 -> 管理面/数据面 | 本机 -> WireGuard -> compute-1 -> 实验网 | 是 |

WireGuard 只解决“人的电脑如何进入实验网”。平台部署、任务执行、容器通信、路由系统下发路径都不应该依赖人的 WireGuard。

## 2. 对 compute 正常路由的影响

当前方案不会影响 compute-1 正常业务路由，原因如下：

- compute-1 没有被改默认路由，也没有给业务网段添加替代路由。
- WireGuard 只新增 `wg-lab` 虚拟接口，用于接收本机客户端流量。
- NAT 规则只匹配 WireGuard 客户端源地址：`10.88.0.0/24` 和 `fd88:88:88::/64`。
- compute 节点自身、业务容器、Node Agent 的正常流量源地址不是 `10.88.0.0/24` 或 `fd88:88:88::/64`，不会命中这些 NAT 规则。
- `ip_forward` / IPv6 forwarding 只是允许 compute-1 转发 WG 客户端流量，不会改变 compute-1 自己访问管理面或数据面的选路。

当前 compute-1 上的规则边界：

```bash
iptables  -t nat -A POSTROUTING -s 10.88.0.0/24      -d 172.16.0.0/24 -o enp7s0           -j MASQUERADE
ip6tables -t nat -A POSTROUTING -s fd88:88:88::/64   -d 3000::/8       -o enx9c69d3680773 -j MASQUERADE
```

安全原则：不要把客户端 `AllowedIPs` 写成 `0.0.0.0/0` 或 `::/0`；不要为了临时排障给 compute 节点批量添加静态路由。

## 3. 网络口径

正式验收阶段：

- 管理面使用 `172.16.0.0/24`。
- 管理节点当前为 `172.16.0.254/24`，只有管理面，没有数据面。
- `compute-1~3` 同时具备管理面、数据面 IPv6，并临时保留 `10.112` 校园网维护入口。
- 终端节点 `h1~h13` 使用 `172.16.0.151-163/24` 管理面和 `3012/3028/3029` 数据面 IPv6。
- `10.112` 只作为当前远程维护和备用入口，正式验收不要依赖它。

字段映射：

| 字段/配置 | 当前调试 | 验收口径 | 用途 |
| --- | --- | --- | --- |
| `nodes.management_ip` | 10.112 IPv4 | 172.16 IPv4 | Manager 调 Node Agent、SSH 运维 |
| `nodes.agent_address` | `http://10.112.x.x:8001` | `http://172.16.x.x:8001` | 控制面 API |
| `nodes.business_ipv6` | 校园网/临时 IPv6 | 数据面 IPv6 | 容器互访、路由系统选路 |
| `MANAGER_PUBLIC_URL` | `http://10.112.244.94:8181` | `http://172.16.0.254:8181` | worker 回调 Manager |
| `PRIVATE_REGISTRY` | `10.112.244.94:5000` | `172.16.0.254:5000` | 镜像推拉 |
| `PREFER_BUSINESS_IPV6` | `true` | `true` | 优先注入 IPv6 PEER URL |

详细节点清单见 [验收网络配置说明](/Users/yanjia/codes/manage_deploy/docs/deployment/验收网络配置说明.md)。管理面/数据面两两互通检测和 IP 更新流程见 [验收网络互通检测与IP更新流程](/Users/yanjia/codes/manage_deploy/docs/deployment/验收网络互通检测与IP更新流程.md)。

## 4. WireGuard 基线

compute-1 是固定 WireGuard server：

```text
server config: /etc/wireguard/wg-lab.conf
listen port:   51820
server wg ip:  10.88.0.1/24, fd88:88:88::1/64
```

本地客户端配置放在 ignored 目录，不提交密钥：

```text
Mac campus: /Users/yanjia/codes/manage_deploy/ops/secrets/wglabmac.conf
Mac mgmt:   /Users/yanjia/codes/manage_deploy/ops/secrets/wglabmgmt.conf
Windows:    /Users/yanjia/codes/manage_deploy/ops/secrets/wg-lab-windows.conf
```

客户端选择规则：

| 场景 | 配置文件 | Endpoint | AllowedIPs |
| --- | --- | --- | --- |
| Mac 未接管理网，走校园网到 compute-1 | `wglabmac.conf` | `10.112.38.25:51820` | `172.16.0.0/24, 3012::/16, 3028::/16, 3029::/16` |
| Mac 已接管理网 | `wglabmgmt.conf` | `172.16.0.101:51820` | `3012::/16, 3028::/16, 3029::/16` |
| Windows 已接管理网 | `wg-lab-windows.conf` | `172.16.0.101:51820` | `3012::/16, 3028::/16, 3029::/16` |

如果电脑已经直连 `172.16` 管理网，就不要再把 `172.16.0.0/24` 放进 WireGuard `AllowedIPs`，避免和本机物理网卡路由重叠。

## 5. 客户端使用

Mac 启动。macOS `wg-quick` 要求配置文件名是合法接口名，所以使用 `wglabmac.conf` 或 `wglabmgmt.conf`，不要用带连字符的文件名：

```bash
cd /Users/yanjia/codes/manage_deploy
sudo /opt/homebrew/bin/wg-quick up ./ops/secrets/wglabmac.conf
```

Mac 停止：

```bash
cd /Users/yanjia/codes/manage_deploy
sudo /opt/homebrew/bin/wg-quick down ./ops/secrets/wglabmac.conf
```

Windows 使用图形客户端：

```text
WireGuard -> Import tunnel(s) from file -> 选择 wg-lab-windows.conf -> Activate
```

如果 Windows 能 `ping` 数据面 IPv6，但浏览器打不开页面，优先检查其它 VPN、代理、浏览器安全软件。2026-06-21 已确认关闭其它 VPN 后，Windows 可以访问数据面 IPv6 HTTP 页面。

## 6. 验证方法

本机连上 WireGuard 后验证：

```bash
ping6 3012:9::fc36:9ea:22bf:a73
curl -g -sS 'http://[3012:3::6e49:e736:feb7:22cd]:19191/'
```

compute-1 上查看 server 状态：

```bash
ssh -p 2345 chengyubin@10.112.38.25 'sudo wg show wg-lab'
```

只读检查 NAT 规则边界：

```bash
ssh -p 2345 chengyubin@10.112.38.25 \
  'sudo iptables -t nat -S POSTROUTING | grep 10.88; sudo ip6tables -t nat -S POSTROUTING | grep fd88'
```

全量验收仍然以互通矩阵为准：

```bash
python3 ops/network/acceptance/check_connectivity.py \
  --plane management \
  --matrix \
  --source-scope plane \
  --source-profile acceptance \
  --timeout 1

python3 ops/network/acceptance/check_connectivity.py \
  --plane data \
  --matrix \
  --source-scope plane \
  --source-profile acceptance \
  --timeout 1
```

临时 HTTP 探针只用于确认本机经 WG 能访问数据面 IPv6，不属于项目部署。确认不需要后可在 compute-1 清理：

```bash
ssh -p 2345 chengyubin@10.112.38.25 \
  'test -f /tmp/manage_deploy_ipv6_probe.pid && kill $(cat /tmp/manage_deploy_ipv6_probe.pid) || true'
```

## 7. 服务端维护

启动 compute-1 WireGuard：

```bash
ssh -p 2345 chengyubin@10.112.38.25 'sudo wg-quick up wg-lab'
```

停止 compute-1 WireGuard：

```bash
ssh -p 2345 chengyubin@10.112.38.25 'sudo wg-quick down wg-lab'
```

当前不强制设置开机自启。若验收现场需要长期保留，应先确认 compute-1 的 `172.16.0.101/24` 已持久化，再启用：

```bash
ssh -p 2345 chengyubin@10.112.38.25 'sudo systemctl enable wg-quick@wg-lab'
```

注意：如果 compute-1 重启后 `172.16.0.101/24` 丢失，WireGuard 仍可能通过 `10.112.38.25` 握手，但访问 `172.16.0.0/24` 会失败。需要先恢复 `enp7s0=172.16.0.101/24`。

## 8. 验收迁移时必须统一的配置

迁移时 Manager、Node Agent、Registry、MySQL、MinIO 都走 `172.16` 管理网：

```text
DATABASE_URL=mysql+aiomysql://root:<password>@172.16.0.254:3306/task_manager
MINIO_ENDPOINT=http://172.16.0.254:9000
MANAGER_PUBLIC_URL=http://172.16.0.254:8181
PRIVATE_REGISTRY=172.16.0.254:5000
PREFER_BUSINESS_IPV6=true
NETWORK_PROFILE=acceptance
```

节点表必须同步：

```text
management_ip  = 172.16 管理面 IPv4
agent_address  = http://<172.16管理IP>:8001
business_ipv6  = 数据面 IPv6
```

每台 compute 和 h1-h13 的 Docker daemon 都必须允许管理节点私有仓库：

```bash
sudo python3 /home/bupt/manage_deploy/scripts/update_docker_insecure_registry.py \
  --registry 172.16.0.254:5000
sudo systemctl restart docker
```

不要用 `python ... | sudo tee /etc/docker/daemon.json` 这类方式覆盖 Docker 配置文件，避免把 `data-root`、NVIDIA runtime、registry mirror 等字段截断丢失。

任务模板中的镜像地址也要统一改为：

```text
172.16.0.254:5000/scientific-matmul:dev
172.16.0.254:5000/scientific-matmul-endpoint:dev
172.16.0.254:5000/low-latency-video:dev
172.16.0.254:5000/low-latency-video-endpoint:dev
```

## 9. SSH 的定位

SSH 仍然保留，但只用于两类场景：

- 登录机器做运维检查。
- WireGuard 不可用时，临时对单个服务做 `ssh -L` 端口转发救急。

SSH 不再作为本机访问实验网的推荐主路径，也不作为业务容器互通路径。业务容器之间仍然必须走系统注入的 `PEER_*_URL=http://[业务IPv6]:端口`。

## 10. 常见问题

**验收电脑已经接了管理网线，还需要 WireGuard 吗？**

只看前端、后端 API、Node Agent、Registry、MinIO 时可以不需要。需要从本机访问数据面 IPv6 服务、多个容器 UI 或做长期调试时，使用 compute-1 WireGuard。

**跳板访问容器 UI 会不会破坏随路计算？**

不会。WireGuard 只是人的访问入口。业务容器 source -> compute -> sink 仍然走数据面 IPv6 和路由系统建立的路径。

**路由同学需要关心 WireGuard 吗？**

一般不需要。WireGuard 是运维和演示访问方式。路由系统只需要关心数据库里的拓扑节点 ID、业务面 IPv6、端口、模态和策略，并按协议回写 placements / network-ready。

**10.112 还能不能用？**

可以作为当前开发和远程维护入口继续保留，但正式验收不要依赖。所有系统配置、数据库字段、镜像地址和验收脚本都应以 `172.16` 管理面和数据面 IPv6 为准。
