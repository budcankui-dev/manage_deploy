# 管理节点跳板与网络迁移方案

本文记录当前调试网络、正式验收网络和 compute-1 跳板接入方案。目标是把“系统运行链路”和“人的调试入口”分清楚：系统运行时 Manager、Node Agent、Registry、MySQL、MinIO 统一走 `172.16` 管理网；人的电脑需要同时访问管理面和数据面时，固定使用 compute-1 做跳板/网关。

本文只描述方案，不执行迁移。

## 1. 网络口径

当前开发调试阶段：

- 本机可以访问管理节点和部分拓扑节点的 10.112 IPv4。
- 管理节点、计算节点、终端节点都同时有 10.112 IPv4 和对应校园网 IPv6。
- 业务数据面暂时使用校园网 IPv6 模拟，`PREFER_BUSINESS_IPV6=true` 时容器之间通过 `http://[IPv6]:port` 通信。

正式验收阶段：

- 管理面从 10.112 IPv4 切换为 172.16 网段，Manager、Node Agent、MySQL、MinIO、Registry 全部走管理面。
- 管理节点只有管理面，当前验收地址为 `172.16.0.254/24`。
- `compute-1~3` 同时具备 `172.16` 管理面、数据面 IPv6，并临时保留 `10.112` 校园网维护入口；正式验收不要依赖 `10.112`。
- 人的电脑如果只看管理服务，直插管理网即可；如果还要同时访问数据面，则使用 compute-1 作为固定跳板/网关。
- 业务数据面使用验收环境分配的独立 IPv6 网段，由路由系统基于 `nodes.business_ipv6` 和端口建立路径。

正式验收地址汇总见 [验收网络配置说明](/Users/yanjia/codes/manage_deploy/docs/deployment/验收网络配置说明.md)。

管理面/数据面两两互通检测、IPv6 地址更新和禁止擅自修改路由表的边界见 [验收网络互通检测与IP更新流程](/Users/yanjia/codes/manage_deploy/docs/deployment/验收网络互通检测与IP更新流程.md)。跳板只解决“人的电脑如何进入实验网”，不等于系统网络已经满足两两互通；验收前仍必须按该流程跑矩阵。

系统设计上必须保持：

| 字段 | 当前调试 | 验收切换后 | 用途 |
|------|----------|------------|------|
| `nodes.management_ip` | 10.112 IPv4 | 172.16 IPv4 | Manager 调 Node Agent、SSH 运维 |
| `nodes.agent_address` | `http://10.112.x.x:8001` | `http://172.16.x.x:8001` | 控制面容器生命周期 API |
| `nodes.business_ip` | 10.112 IPv4，可保留辅助显示 | 可为空或填业务 IPv4 | 非 IPv6 调试兜底 |
| `nodes.business_ipv6` | 校园网 IPv6 | 数据面 IPv6 | 业务容器通信和路由系统选路 |
| `MANAGER_PUBLIC_URL` | `http://10.112.244.94:8181` | `http://<管理节点172.16地址>:8181` | worker 指标回写 Manager |
| `PREFER_BUSINESS_IPV6` | `true` | `true` | 优先向容器注入 IPv6 PEER URL |

不要把 10.112 写成协议前提。10.112 只是当前调试网络，正式验收时应只作为历史示例或临时回退。

## 2. 目标方案总览

系统运行链路和人的接入链路分开设计：

| 链路 | 责任 | 正式地址口径 | 是否依赖跳板 |
| --- | --- | --- | --- |
| Manager -> MySQL | 后端数据库 | `172.16.0.254:3306` | 否 |
| Manager -> MinIO | 结果归档/对象存储 | `172.16.0.254:9000` | 否 |
| Manager / 节点 -> Registry | 镜像推拉 | `172.16.0.254:5000` | 否 |
| Manager -> Node Agent | 部署、停止、删除容器 | `http://<节点172.16管理IP>:8001` | 否 |
| 容器 -> Manager | 业务状态/结果回调 | `MANAGER_PUBLIC_URL=http://172.16.0.254:8181` | 否 |
| 业务容器 -> 业务容器 | source -> compute -> sink 数据流 | `http://[数据面IPv6]:端口` | 否 |
| 人的电脑 -> 管理面 | 浏览前端、API、Agent、Registry 排查 | 直连管理网或 compute-1 隧道 | 可选 |
| 人的电脑 -> 数据面 | 查看业务容器 UI、抓包、临时调试 | 经 compute-1 隧道/端口转发 | 是 |

compute-1 的定位固定为“人的接入跳板/网关”，不承担系统运行依赖。它的后端可达网段保持不变：

```text
172.16.0.0/24         管理面
3012::/16             数据面一组
3028::/16             数据面一组
3029::/16             数据面一组
```

compute-1 的入口地址按场景切换：

| 场景 | 本机入口 | 用途 |
| --- | --- | --- |
| 开发/备用 | `10.112.38.25:2345` | 本机还没有接管理网线时，通过校园网进 compute-1 |
| 验收/正式 | `172.16.0.101:2345` | 本机接入管理网后，通过管理网进 compute-1 |

因此，compute-1 上的 WireGuard/转发/NAT 规则原则上保持一套；切换开发/验收时，只改本机客户端连接 compute-1 的 endpoint。

## 4. 推荐 SSH 配置

SSH 配置建议拆成两类：系统运维登录管理节点，以及人的电脑通过 compute-1 进入实验网。

管理节点 alias 用于登录总控和查看系统服务：

```sshconfig
Host manage-admin
  HostName 10.112.244.94
  User bupt
  Port 22
  IdentityFile ~/.ssh/manage_deploy_test_lab_ed25519
```

compute-1 准备两个入口 alias，后端跳板能力相同，只是本机连入地址不同：

```sshconfig
Host jump-compute1-campus
  HostName 10.112.38.25
  User chengyubin
  Port 2345
  IdentityFile ~/.ssh/manage_deploy_test_lab_ed25519

Host jump-compute1-mgmt
  HostName 172.16.0.101
  User chengyubin
  Port 2345
  IdentityFile ~/.ssh/manage_deploy_test_lab_ed25519
```

开发期没有管理网线时，用 `jump-compute1-campus`；验收期本机接入管理网后，用 `jump-compute1-mgmt`。

拓扑节点建议按验收管理面地址配置，并通过 compute-1 跳转。例如：

```sshconfig
Host topo-compute-1-mgmt
  HostName 172.16.0.101
  User chengyubin
  Port 2345
  IdentityFile ~/.ssh/manage_deploy_test_lab_ed25519
  ProxyJump jump-compute1-mgmt

Host topo-compute-1-campus
  HostName 172.16.0.101
  User chengyubin
  Port 2345
  IdentityFile ~/.ssh/manage_deploy_test_lab_ed25519
  ProxyJump jump-compute1-campus
```

终端节点同理：

```sshconfig
Host topo-h1-mgmt
  HostName 172.16.0.151
  User switchpc1
  Port 22
  ProxyJump jump-compute1-mgmt

Host topo-h1-campus
  HostName 172.16.0.151
  User switchpc1
  Port 22
  ProxyJump jump-compute1-campus
```

如果本机已经直插管理网，并且只访问 `172.16.0.0/24`，终端节点也可以不经过 ProxyJump 直接登录：

```sshconfig
Host topo-h1
  HostName 172.16.0.151
  User switchpc1
  Port 22
```

推荐实践：

- 开发期通过校园网入口：`ProxyJump jump-compute1-campus`。
- 验收期通过管理网入口：`ProxyJump jump-compute1-mgmt`。
- 拓扑节点的 `HostName` 尽量固定写 `172.16` 管理面地址，不再写 `10.112`。
- 需要脚本批量登录 h1-h13 时，根据 `ops/inventory/topology_nodes.json` 生成 alias 或直接拼 `ssh -J` 命令。

这样切换时不需要改所有目标节点，只需要换 compute-1 的入口 alias。

## 5. 当前跳板方式是什么

当前本机 `~/.ssh/config` 已有 `manage-admin`、`manage-compute-1/2/3`，但 `manage-compute-*` 仍是本机直连 10.112 地址。也就是说：

- 现在开发环境下，本机直连 compute 节点还能用。
- 已经验证过本机能通过校园网访问 compute-1，也验证过 compute-1 能访问终端数据面 IPv6。
- 后续建议新增 `jump-compute1-campus` / `jump-compute1-mgmt` 两套入口，并让人的调试流量统一经 compute-1。

这不是修改系统业务流，只是修改运维访问方式。Manager 调 Node Agent 仍然按 `nodes.agent_address` 走管理面；业务容器通信仍然按 `nodes.business_ipv6` 走数据面。

## 6. compute-1 作为本机跳板

compute-1 是固定跳板，本机访问实验网有两种入口：

| 模式 | 本机到 compute-1 | 适用场景 | 推荐程度 |
| --- | --- | --- | --- |
| 验收模式 | 本机接管理网线，访问 `compute-1` 的 `172.16.0.101` | 正式验收、稳定演示 | 推荐 |
| 开发/备用模式 | 本机走校园网，访问 `compute-1` 的 `10.112.38.25` | 尚未接管理网线、远程调试 | 可用但不要作为验收假设 |

`compute-1` 同时在 `172.16` 管理面和数据面 IPv6 上，因此适合做本机进入实验网络的入口。这里的跳板只解决“人的电脑如何访问两个网段”，不改变平台运行时路径：

- Manager 访问 Node Agent 仍走 `172.16` 管理面。
- 私有 Registry、MySQL、MinIO、后端 API 在验收时都使用管理节点 `172.16.0.254`。
- 业务容器之间的 source -> compute -> sink 仍走数据面 IPv6。

### 6.1 SSH 方案

SSH 适合快速排障和少量端口访问，配置最少，不需要修改系统路由。

访问管理面服务：

```bash
ssh -N -L 18181:172.16.0.254:8181 chengyubin@172.16.0.101 -p 2345
curl -sS http://127.0.0.1:18181/health
```

开发/备用模式下，把入口地址换成 `10.112.38.25`：

```bash
ssh -N -L 18181:172.16.0.254:8181 chengyubin@10.112.38.25 -p 2345
```

访问某个数据面 IPv6 容器 UI：

```bash
ssh -N -L 19100:[3012:a::ec4:7aff:fe85:7815]:9100 chengyubin@172.16.0.101 -p 2345
```

开发/备用模式下同样只换入口地址：

```bash
ssh -N -L 19100:[3012:a::ec4:7aff:fe85:7815]:9100 chengyubin@10.112.38.25 -p 2345
```

SSH 的优点是安全、简单、不会误把本机流量导进实验网；缺点是每个服务都要开一个转发端口，不适合浏览多个节点和动态端口。

### 6.2 WireGuard 方案

如果希望本机像接入实验网一样访问两个网段，WireGuard 是更合适的方案。建议把 `compute-1` 作为 WireGuard server，本机作为 client：

```text
本机 PC
  -> WireGuard 隧道
  -> compute-1
      -> 172.16.0.0/24 管理面
      -> 3012/3028/3029 数据面 IPv6
```

推荐把 compute-1 上的 WireGuard server 固定成一套，只在本机客户端切换 endpoint：

| 场景 | WireGuard Endpoint |
| --- | --- |
| 开发/备用 | `10.112.38.25:51820` |
| 验收/正式 | `172.16.0.101:51820` |

推荐路由策略：

- WireGuard 只用于本机运维访问，不承载业务容器真实数据流。
- `AllowedIPs` 只放实验网段，不使用 `0.0.0.0/0` 或 `::/0`。
- 管理面 IPv4 推荐走 NAT/masquerade，避免给所有节点加回程路由。
- 数据面 IPv6 优先只转发必要端口；若要全量访问数据面 IPv6，需要现场确认 IPv6 转发、回程路由或 NAT66/NPTv6 策略。

示例网段规划：

```text
WireGuard IPv4: 10.88.0.0/24
compute-1 wg0: 10.88.0.1
本机 wg0: 10.88.0.2
本机 AllowedIPs: 172.16.0.0/24, 3012::/16, 3028::/16, 3029::/16
```

compute-1 需要开启转发：

```bash
sudo sysctl -w net.ipv4.ip_forward=1
sudo sysctl -w net.ipv6.conf.all.forwarding=1
```

管理面 IPv4 可用 NAT 让回包回到 compute-1：

```bash
sudo iptables -t nat -A POSTROUTING -s 10.88.0.0/24 -d 172.16.0.0/24 -j MASQUERADE
```

如果只需要访问管理面，做到这里通常就够了。数据面 IPv6 建议先用 SSH 转发或在 compute-1 上 curl 验证；只有确实需要本机浏览器直连多个数据面服务时，再配置 IPv6 路由/NAT，并记录现场规则。

WireGuard 验证命令：

```bash
ping 172.16.0.254
curl -sS http://172.16.0.254:8181/health
curl -sS http://172.16.0.101:8001/health
ping -6 3012:a::ec4:7aff:fe85:7815
```

选择建议：

- 少量端口、临时排障：用 SSH 转发。
- 本机需要稳定访问 `172.16` 管理面多个服务：用 WireGuard。
- 本机还要访问多个数据面 UI：优先 WireGuard，但先确认 IPv6 路由策略，别在验收当天临时开全网转发。

## 7. 跳板连通性验证

开发期先验证本机经校园网到 compute-1：

```bash
ssh jump-compute1-campus "hostname; ip -br addr"
```

验收期先验证本机经管理网到 compute-1：

```bash
ssh jump-compute1-mgmt "hostname; ip -br addr"
```

再验证经 compute-1 到管理面节点：

```bash
ssh -J jump-compute1-mgmt switchpc1@172.16.0.151 "hostname; ip -br addr"
ssh -J jump-compute1-mgmt switchpc1@172.16.0.163 "hostname; ip -br addr"
```

开发期使用校园网入口时，只改 `-J`：

```bash
ssh -J jump-compute1-campus switchpc1@172.16.0.151 "hostname; ip -br addr"
```

这些命令验证的是人的跳板链路。系统运行链路仍然要在管理节点上验证 Manager 到各 Agent、MySQL、MinIO、Registry 的 `172.16` 连通性。

## 9. 通过跳板访问业务容器 IPv6 UI

业务容器 UI 和容器间通信应走数据面 IPv6。由于本机可能没有到数据面 IPv6 的路由，浏览器访问容器 UI 时使用 compute-1 作为跳板做端口转发。

示例：通过验收入口访问 h5 上业务 UI `9100`：

```bash
ssh -N -L 19100:[3012:a::250:56ff:fe8b:7d13]:9100 chengyubin@172.16.0.101 -p 2345
```

开发/备用入口只换 compute-1 地址：

```bash
ssh -N -L 19100:[3012:a::250:56ff:fe8b:7d13]:9100 chengyubin@10.112.38.25 -p 2345
```

然后本机浏览器打开：

```text
http://127.0.0.1:19100/
```

说明：

- `-L 本地端口:[IPv6]:远端端口` 中 IPv6 必须用方括号。
- receiver 首页默认展示最近一次工单，页面顶部“切换已接收工单”下拉可切换历史工单；按工单 ID 排查时使用 `/latest` 或 `/orders/{order_id}` JSON 接口。
- 该方式只解决“本机看不到数据面 IPv6”的浏览器访问问题，不改变容器真实通信路径。
- 容器之间仍应使用系统注入的 `PEER_*_URL=http://[业务IPv6]:动态端口`。

如果 SSH 客户端对 IPv6 转发兼容性不好，可以先登录 compute-1 再从 compute-1 发起 curl 验证：

```bash
ssh jump-compute1-mgmt "curl -g -sS 'http://[3012:a::250:56ff:fe8b:7d13]:9100/health'"
```

## 10. 已实测的 SSH 跳板基线

2026-06-20 已用开发/备用入口 `10.112.38.25:2345` 跑通 compute-1 作为跳板，不涉及项目部署和迁移。

本机启动管理面端口转发：

```bash
ssh -N -f \
  -o ExitOnForwardFailure=yes \
  -o ServerAliveInterval=30 \
  -o ServerAliveCountMax=3 \
  -L 18182:172.16.0.254:8181 \
  -L 18002:172.16.0.101:8001 \
  -p 2345 chengyubin@10.112.38.25
```

验证：

```bash
curl -sS http://127.0.0.1:18182/health
curl -sS http://127.0.0.1:18002/health
```

期望结果：

```text
{"status":"healthy"}
{"status":"healthy"}
```

数据面如果暂时没有业务容器端口可测，可以只启动临时网络探针验证链路。该探针只用于跳板网络验证，不属于项目部署：

```bash
ssh -p 2345 chengyubin@10.112.38.25 \
  "nohup python3 -m http.server 19191 --bind 3012:3::6e49:e736:feb7:22cd --directory /tmp >/tmp/manage_deploy_data_probe.log 2>&1 & echo \$! >/tmp/manage_deploy_data_probe.pid"
```

本机启动数据面 IPv6 转发：

```bash
ssh -N -f \
  -o ExitOnForwardFailure=yes \
  -L '19193:[3012:3::6e49:e736:feb7:22cd]:19191' \
  -p 2345 chengyubin@10.112.38.25
```

验证：

```bash
curl -sS -I http://127.0.0.1:19193/ | head -n 1
```

期望结果：

```text
HTTP/1.0 200 OK
```

清理本机 SSH 隧道：

```bash
lsof -nP -iTCP:18182 -iTCP:18002 -iTCP:19193 -sTCP:LISTEN
kill <对应 ssh 进程 PID>
```

清理 compute-1 临时探针：

```bash
ssh -p 2345 chengyubin@10.112.38.25 \
  "test -f /tmp/manage_deploy_data_probe.pid && kill \$(cat /tmp/manage_deploy_data_probe.pid) || true"
```

验收期把上面命令中的入口 `10.112.38.25` 换成 `172.16.0.101`；目标地址仍使用 `172.16.0.254`、`172.16.0.101` 和数据面 IPv6。

## 12. 切换到验收网络时要改什么

2026-06-21 已在当前管理节点执行过一次验收 profile 切换：数据库已备份，`nodes` 表已更新为 `172.16` 管理面和 `3012/3028/3029` 数据面 IPv6，两个业务模板镜像已更新为 `172.16.0.254:5000/...`，后端已用 `NETWORK_PROFILE=acceptance` 启动。后续现场迁移或回滚时仍按本节顺序执行，优先改配置和数据库，不改业务代码。

执行前必须先备份数据库：

```bash

```

1. 更新管理节点 `.env`：


迁移顺序必须先处理管理节点基础组件，再更新节点表和任务模板：

- MinIO 仍走管理面，`MINIO_ENDPOINT=http://172.16.0.254:9000`。当前管理节点复用原 `minio` 容器，API 为 `9000`，控制台为 `9002`。
- 私有 Registry 统一走 `172.16.0.254:5000`，复用原 `registry:2` 容器，不新建第二套仓库。所有节点 Docker daemon 都必须配置该 insecure registry，同时开发期可保留 `10.112.244.94:5000` 作为回退。
- 后端脚本默认值由 `NETWORK_PROFILE=acceptance` 控制；显式 `MANAGER_API_BASE`、`PRIVATE_REGISTRY`、`MINIO_ENDPOINT` 仍可覆盖。

3. 更新私有镜像仓库配置：

- 管理节点上的私有 Registry 容器必须对外使用管理面地址，例如 `172.16.0.254:5000`。

## 13. 验收前必须确认的现象

- 系统运行链路不依赖 `10.112`：Manager、Node Agent、Registry、MySQL、MinIO 都使用 `172.16` 管理网。
- 人的电脑开发期可以通过 `10.112.38.25` 进入 compute-1，验收期可以通过 `172.16.0.101` 进入 compute-1。
- compute-1 的跳板后端目标网段不随入口变化：`172.16.0.0/24` 和数据面 IPv6。
- 管理节点能访问所有 Node Agent 的管理面地址。
- compute-1 能访问需要调试的业务数据面 IPv6。
- 工单 DAG 同时包含拓扑节点 ID、节点别名、管理面部署结果、业务面 IPv6 和业务端口。
- 路由系统拿到的是数据面 IP/IPv6 和端口，不依赖 10.112 校园网。
- 容器内 `PEER_*_URL` 是 IPv6 URL，形如 `http://[3012:...]:18000`、`http://[3028:...]:18000` 或 `http://[3029:...]:18000`。
- 本机查看容器 UI 时可以通过 compute-1 SSH 本地转发或 WireGuard 访问，不要求本机有数据面 IPv6 直连路由。
- 所有节点 `docker info` 中的 insecure registry 已包含 `172.16.0.254:5000`，任务模板镜像不再指向 `10.112.244.94:5000`。

## 14. 常见问题

**compute-1 重启 Docker 后失联怎么办？**

先把它视为节点网络/firewalld 问题，不要继续对其它节点批量重启 Docker。2026-06-21 排查到的典型根因是 firewalld/NetworkManager 把 `docker0` 放入 `public` zone，Docker 启动时报 `ZONE_CONFLICT: 'docker0' already bound to 'public'`。如果 SSH、Node Agent、ping 同时不通，需要本机控制台或机房带外入口执行：

```bash
sudo nft flush ruleset
sudo systemctl stop firewalld
sudo systemctl reset-failed docker docker.socket
sudo systemctl start docker
curl -sS http://127.0.0.1:8001/health
```

恢复后再检查 `10.112.38.25:2345`、`172.16.0.101:8001` 和数据面 IPv6。这个恢复步骤只用于救援；正式迁移前应把 compute-1 的 `172.16.0.101/24` 和 Docker/firewalld 关系持久化。

**为什么不让本机直接访问所有节点？**

正式验收时，本机可能只接入管理网，能访问 `172.16`，但不一定具备数据面 IPv6 路由。提前使用 compute-1 跳板模式，可以同时覆盖管理面和数据面调试，减少切网后的故障。

**跳板访问容器 UI 会不会破坏随路计算？**

不会。SSH 本地端口转发只用于浏览器查看结果页面。业务容器之间的 source -> compute -> sink 通信仍然走数据面 IPv6 和路由系统建立的路径。

**10.112 IPv4 和校园网 IPv6 现在还能不能用？**

可以，当前开发调试继续使用它们。但文档和代码不要把它们当成正式验收假设；正式切换时用 172.16 管理面和新的数据面 IPv6 替代。

**验收时电脑已经接了管理网线，还需要 WireGuard 吗？**

只看前端、后端 API、Node Agent、Registry、MinIO 时可以不需要，直接访问 `172.16` 即可。需要从本机访问数据面 IPv6 服务、多个容器 UI 或做长期调试时，再使用 compute-1 的 WireGuard。

**路由同学需要关心跳板机吗？**

一般不需要。跳板机是我们运维和演示访问方式。路由同学只需要读取数据库中的拓扑节点 ID、业务面 IPv6、端口、模态和策略，并按协议回写 placements / network-ready。
