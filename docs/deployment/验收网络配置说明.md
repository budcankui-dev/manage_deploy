# 验收网络配置说明

本文整合验收时的管理网段、数据平面网段和临时校园网信息。正式验收时不要依赖 `10.112` 网段；它只作为当前远程维护和兜底入口，现场不保证可用。

## 网络分层

| 网络 | 地址段 | 用途 | 备注 |
| --- | --- | --- | --- |
| 管理面 | `172.16.0.0/24` | Manager 调 Node Agent、SSH 运维、MySQL、MinIO、Registry、前端/后端 API | 正式验收主控制网络 |
| 数据面 | `3012:*` / `3028:*` / `3029:*` IPv6 | 业务容器之间通信、路由系统下发路径或流表 | 写入 `nodes.business_ipv6` |
| 临时校园网 | `10.112.*` | 当前远程维护、批量配置入口 | 验收时不保证可用 |

系统字段映射：

| 平台字段/配置 | 验收时应填写 |
| --- | --- |
| `nodes.management_ip` | `172.16` 管理面 IPv4 |
| `nodes.agent_address` | `http://<172.16管理IP>:8001` |
| `nodes.business_ip` | 可填管理面 IPv4 作为展示/兜底，但业务主链路不依赖它 |
| `nodes.business_ipv6` | 数据面 IPv6，容器互访优先使用 |
| `MANAGER_PUBLIC_URL` | `http://172.16.0.254:8181` |
| `MINIO_ENDPOINT` | `http://172.16.0.254:9000` |
| `DATABASE_URL` | 指向 `172.16.0.254:3306` |
| 私有 Registry | `172.16.0.254:5000` |
| `PREFER_BUSINESS_IPV6` | `true` |

本机访问实验网络的固定方案：

- 稳定开发/验收：在 `compute-1` 上使用 WireGuard，让本机只路由 `172.16.0.0/24` 和必要的 `3012/3028/3029` 数据面地址。详细方案见 [管理节点跳板与网络迁移方案](/Users/yanjia/codes/manage_deploy/docs/deployment/管理节点跳板与网络迁移方案.md)。
- 临时排障：WireGuard 不可用时，才通过 `compute-1` 做单服务 SSH 端口转发救急；不再作为常规路径。
- 管理面/数据面两两互通检测、IPv6 地址更新和脚本边界见 [验收网络互通检测与IP更新流程](/Users/yanjia/codes/manage_deploy/docs/deployment/验收网络互通检测与IP更新流程.md)。

## 配置脚本

相关脚本已收在 `ops/network/acceptance/`：

| 脚本 | 用途 |
| --- | --- |
| `check_connectivity.py` | 从本机或指定 SSH 源节点检查管理面/数据面连通性，可复用排查全网可达性 |
| `config_ens224.ps1` | 通过临时 `10.112` SSH 到 h1-h13，配置 `ens224` 为 `172.16.0.151-163/24` |
| `verify_ens224.ps1` | 验证 h1-h13 的 `ens224` 是否已有预期管理面 IPv4 |
| `reset_ens192_ipv6.ps1` | 当终端节点没有数据面 IPv6 时，重置 `ens192` 并触发重新获取 IPv6 |

脚本使用 `switchpc1@10.112.*` 作为临时入口。验收现场如果 `10.112` 不可用，应通过管理节点或现场控制台执行等价命令。

连通性检查示例：

```bash
# 本机已有 WireGuard/管理网直连时，检查全部管理面和数据面。
python3 ops/network/acceptance/check_connectivity.py --plane both

# 从 compute-1 作为源节点检查管理面，期望 17/17 全通。
python3 ops/network/acceptance/check_connectivity.py \
  --plane management \
  --ssh '-p 2345 chengyubin@10.112.38.25'

# 从 compute-1 作为源节点检查数据面。
python3 ops/network/acceptance/check_connectivity.py \
  --plane data \
  --ssh '-p 2345 chengyubin@10.112.38.25'

# 验收现场全量两两互通源预览，要求本机已接管理网或 WireGuard 已连通。
python3 ops/network/acceptance/check_connectivity.py \
  --plane both \
  --matrix \
  --source-scope plane \
  --source-profile acceptance \
  --list-sources

# 验收现场全量管理面矩阵：17 个源到 17 个管理面目标。
python3 ops/network/acceptance/check_connectivity.py \
  --plane management \
  --matrix \
  --source-scope plane \
  --source-profile acceptance \
  --timeout 1

# 验收现场全量数据面矩阵：16 个源到 16 个数据面目标，admin 不参与。
python3 ops/network/acceptance/check_connectivity.py \
  --plane data \
  --matrix \
  --source-scope plane \
  --source-profile acceptance \
  --timeout 1

# h1-h13 未配置免密时，可用本地 ignored 密码文件跑开发/备用入口的全量数据面矩阵。
python3 ops/network/acceptance/check_connectivity.py \
  --plane data \
  --matrix \
  --source-scope plane \
  --source-profile current \
  --password-file ops/secrets/terminal-credentials.local.md \
  --timeout 1
```

已有修复脚本边界：

- h1-h13 管理面 `ens224` IP 丢失或不符合 `172.16.0.151-163/24`：用 `config_ens224.ps1` 修复，再用 `verify_ens224.ps1` 验证。
- h1-h13 数据面 `ens192` 没有 IPv6 或地址异常：可用 `reset_ens192_ipv6.ps1` 重置接口并触发重新获取 IPv6。
- compute-1 管理面 `172.16.0.101/24` 重启后丢失：当前临时恢复命令是 `sudo ip link set enp7s0 up && sudo ip -4 addr flush dev enp7s0 && sudo ip addr add 172.16.0.101/24 dev enp7s0`，暂未做持久化。
- compute 节点数据面与其它数据面节点不通：不要自行修改路由表或用终端脚本修复；应把现象交给网络/路由侧核对数据面端口、RA/路由下发和转发表。任何路由变更都需要先确认再执行。

危险操作提醒：不要在未确认的情况下执行 `ip route add|del`、`ip -6 route add|del`、`ip addr flush`、`iptables`、`sysctl forwarding` 等命令。验收检测默认只做只读探测，修复动作必须单独确认。

单机配置示例：

```bash
sudo ip link set ens224 up
sudo ip addr flush dev ens224
sudo ip addr add 172.16.0.163/24 dev ens224
ip addr show ens224
```

## 验收切换步骤

0. 先确认管理节点基础组件监听管理网：

```bash
curl -sS http://172.16.0.254:8181/health
curl -sS http://172.16.0.254:9000/minio/health/live
curl -sS http://172.16.0.254:5000/v2/_catalog
```

管理节点 MySQL、MinIO、Registry 是验收环境的基础组件中心：

- MySQL 仍是唯一数据库，不新建本地库或第二套验收库。
- MinIO 用管理面提供结果归档服务，业务容器回传结果不走数据面。当前复用管理节点原有 `minio` 容器和 `/mnt/data/minIO` 数据目录，API 端口是 `9000`，Web 控制台端口是 `9002`，避免和 Portainer Agent 的 `9001` 冲突。
- Registry 复用管理节点现有 `registry:2` 容器，不需要新建第二套私有仓库。该容器监听 `0.0.0.0:5000`，所以当前开发地址 `10.112.244.94:5000` 和验收管理网地址 `172.16.0.254:5000` 指向同一个仓库数据。
- Registry 必须用 `172.16.0.254:5000` 对所有 compute 和 h1-h13 可达；开发期可同时保留 `10.112.244.94:5000` 作为回退入口。

3. 所有需要拉镜像的节点更新 Docker insecure registry。

必须保留原有 `daemon.json` 字段，只追加 `172.16.0.254:5000`。推荐在每台节点上使用仓库脚本：

```bash
sudo python3 /home/bupt/manage_deploy/scripts/update_docker_insecure_registry.py \
  --registry 172.16.0.254:5000
sudo systemctl restart docker
docker info | sed -n '/Insecure Registries:/,/Live Restore Enabled:/p'
```

这一步是硬性要求：管理节点上的私有仓库容器对外地址、每台计算节点和终端/虚拟节点的 Docker daemon 配置、任务模板中的镜像地址必须统一切到 `172.16.0.254:5000`。不能继续依赖 `10.112.244.94:5000`，因为验收时 `10.112` 不保证可用。

不要使用 `python ... | sudo tee /etc/docker/daemon.json` 这类命令读取并覆盖同一个文件；`tee` 会先截断目标文件，可能导致 `data-root`、NVIDIA runtime、registry mirror 等字段丢失。管理节点和计算节点 Docker 数据目录一旦切回默认路径，容器会看起来“全部消失”。

compute-1 上曾出现 firewalld/NetworkManager 把 `docker0` 绑定到 `public` zone，Docker 启动时报 `ZONE_CONFLICT: 'docker0' already bound to 'public'`。如果重启 Docker 后 compute-1 失联，需要从本机控制台或带外入口执行：

```bash
sudo nft flush ruleset
sudo systemctl stop firewalld
sudo systemctl reset-failed docker docker.socket
sudo systemctl start docker
curl -sS http://127.0.0.1:8001/health
```

恢复 SSH 后再确认 `10.112.38.25`、`172.16.0.101`、`3012:3::6e49:e736:feb7:22cd` 仍在对应网卡上。
