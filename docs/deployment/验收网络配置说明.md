# 验收网络配置说明

本文整合验收时的管理网段、数据平面网段和临时校园网信息。正式验收时不要依赖 `10.112` 网段；它只作为当前远程维护和兜底入口，现场不保证可用。

## 网络分层

| 网络 | 地址段 | 用途 | 备注 |
| --- | --- | --- | --- |
| 管理面 | `172.16.0.0/24` | Manager 调 Node Agent、SSH 运维、MySQL、MinIO、Registry、前端/后端 API | 正式验收主控制网络 |
| 数据面 | `3012:*` / `3028:*` / `3029:*` IPv6 | 业务容器之间通信、路由系统下发路径或流表 | 写入 `nodes.business_ipv6` |
| 临时校园网 | `10.112.*` | 当前远程维护、批量配置入口 | 验收时不保证可用 |

系统字段映射：

| 平台字段/配置 | 验收时应填写 |
| --- | --- |
| `nodes.management_ip` | `172.16` 管理面 IPv4 |
| `nodes.agent_address` | `http://<172.16管理IP>:8001` |
| `nodes.business_ip` | 可填管理面 IPv4 作为展示/兜底，但业务主链路不依赖它 |
| `nodes.business_ipv6` | 数据面 IPv6，容器互访优先使用 |
| `MANAGER_PUBLIC_URL` | `http://172.16.0.254:8181` |
| `MINIO_ENDPOINT` | `http://172.16.0.254:9000` |
| `DATABASE_URL` | 指向 `172.16.0.254:3306` |
| 私有 Registry | `172.16.0.254:5000` |
| `PREFER_BUSINESS_IPV6` | `true` |

本机访问实验网络推荐两层方案：

- 临时排障：通过 `compute-1` 做 SSH 端口转发，入口可用 `172.16.0.101`，开发备用时可用 `10.112.38.25`。
- 稳定开发/验收：在 `compute-1` 上部署 WireGuard，让本机只路由 `172.16.0.0/24` 和必要的 `3012:*` 数据面地址。详细方案见 [管理节点跳板与网络迁移方案](/Users/yanjia/codes/manage_deploy/docs/deployment/管理节点跳板与网络迁移方案.md)。

## 管理节点与计算节点

管理节点只有管理面，不参与业务数据面；`compute-1` 到 `compute-3` 同时具备管理面、数据面和临时 `10.112` 校园网。

## 配置脚本

相关脚本已收在 `ops/network/acceptance/`：

| 脚本 | 用途 |
| --- | --- |
| `config_ens224.ps1` | 通过临时 `10.112` SSH 到 h1-h13，配置 `ens224` 为 `172.16.0.151-163/24` |
| `verify_ens224.ps1` | 验证 h1-h13 的 `ens224` 是否已有预期管理面 IPv4 |
| `reset_ens192_ipv6.ps1` | 当终端节点没有数据面 IPv6 时，重置 `ens192` 并触发重新获取 IPv6 |

脚本使用 `switchpc1@10.112.*` 作为临时入口。验收现场如果 `10.112` 不可用，应通过管理节点或现场控制台执行等价命令。

单机配置示例：

```bash
sudo ip link set ens224 up
sudo ip addr flush dev ens224
sudo ip addr add 172.16.0.163/24 dev ens224
ip addr show ens224
```

## 验收切换步骤

1. 在管理节点 `.env` 中切换管理面地址：

```text
DATABASE_URL=mysql+aiomysql://root:<password>@172.16.0.254:3306/task_manager
MINIO_ENDPOINT=http://172.16.0.254:9000
MANAGER_PUBLIC_URL=http://172.16.0.254:8181
BACKEND_HOSTNAME=admin
BACKEND_PORT=8181
PREFER_BUSINESS_IPV6=true
```

2. 更新平台节点表：

```text
admin      management_ip=172.16.0.254  agent_address=http://172.16.0.254:8001  node_kind=admin   is_schedulable=false
compute-1 management_ip=172.16.0.101  agent_address=http://172.16.0.101:8001  business_ipv6=<选定的3012:3::...>
compute-2 management_ip=172.16.0.102  agent_address=http://172.16.0.102:8001  business_ipv6=<选定的3012:9::...>
compute-3 management_ip=172.16.0.103  agent_address=http://172.16.0.103:8001  business_ipv6=3012:a::ec4:7aff:fe85:7815
h1-h13    management_ip=172.16.0.151-163 agent_address=http://172.16.0.151-163:8001 business_ipv6=<现场确认的3012:*>
```

3. 所有需要拉镜像的节点更新 Docker insecure registry：

```json
{
  "insecure-registries": ["172.16.0.254:5000"]
}
```

这一步是硬性要求：管理节点上的私有仓库容器对外地址、每台计算节点和终端/虚拟节点的 Docker daemon 配置、任务模板中的镜像地址必须统一切到 `172.16.0.254:5000`。不能继续依赖 `10.112.244.94:5000`，因为验收时 `10.112` 不保证可用。

4. 更新任务模板中的镜像地址，例如：

```text
172.16.0.254:5000/scientific-matmul:dev
172.16.0.254:5000/low-latency-video:dev
172.16.0.254:5000/node-agent:dev
```

5. 验证管理面：

```bash
curl -sS http://172.16.0.254:8181/health
curl -sS http://172.16.0.101:8001/health
curl -sS http://172.16.0.102:8001/health
curl -sS http://172.16.0.103:8001/health
```

6. 验证数据面：

```bash
ping -6 -c 2 3012:3::405e:c4cc:af49:e459
ping -6 -c 2 3012:9::319a:124f:5a8:51e4
ping -6 -c 2 3012:a::ec4:7aff:fe85:7815
```

业务验收时还要进入容器检查：

```bash
env | grep -E 'PEER_.*URL|TASK_PEERS_JSON|MANAGER_API_BASE'
```

期望：

- `MANAGER_API_BASE` 指向 `http://172.16.0.254:8181`。
- `PEER_*_URL` 使用 `http://[3012:...]:端口`。
- sink 日志出现业务完成标记，例如 `SINK_DONE` 或 `VIDEO_SINK_DONE`。
