# 标准化部署与运维流程

本文约定管理节点、计算节点和终端节点的固定目录、启动方式和验证命令。后续更新部署、排查节点资源、跑端到端验证时，以本文为准，避免临时目录、不同 Python 路径和旧进程造成环境漂移。

## 0. 权威口径

后续部署、重启、排障只认本文。其它文档如果出现旧目录、旧 Python 环境或 `git pull` 示例，以本文为准。

### 0.1 固定变量

所有部署命令默认使用下面这组变量。后续新增脚本、文档或交接说明时，也应优先引用这些路径，不要临时发明新目录。

当前变量仍以 `10.112` 调试网络为默认值。正式验收切换时，先按 [验收网络配置说明](/Users/yanjia/codes/manage_deploy/docs/deployment/验收网络配置说明.md) 将管理面切到 `172.16.0.0/24`，数据面切到 `3012:*` IPv6，再替换 `REGISTRY`、`MANAGER_PUBLIC_URL`、`MINIO_ENDPOINT` 和 `nodes` 表地址。`10.112` 只作为临时校园网维护入口，验收现场不保证可用。

验收切换后，私有仓库必须统一使用管理节点管理面地址：


### 0.2 标准部署流水线

每次更新部署按这个顺序执行。除非明确是在排查某个单点问题，否则不要跳步。

```bash
cd /Users/yanjia/codes/manage_deploy
git status --short --branch

# 1. 本地验证。按改动范围选择后端测试；前端只要改了页面就必须 build。
backend/venv/bin/python -m py_compile backend/scripts/e2e_user_endpoint_manual_containers.py
npm --prefix frontend run build

# 2. 同步到管理节点。远端 .env、venv、node_modules、数据库和运行产物不覆盖。
rsync -az --delete \
  --exclude='.git/' \
  --exclude='.DS_Store' \
  --exclude='backend/.env' \
  --exclude='frontend/.env' \
  --exclude='backend/venv/' \
  --exclude='frontend/node_modules/' \
  --exclude='node_modules/' \
  --exclude='backend/*.db' \
  --exclude='backend/.tmp/' \
  --exclude='.pytest_cache/' \
  --exclude='frontend/test-results/' \
  --exclude='frontend/playwright-report/' \
  --exclude='reports/' \
  --exclude='output/' \
  ./ manage-admin:/home/bupt/manage_deploy/

# 3. 重启管理节点后端。只使用固定 Python，不使用 backend/venv。
ssh manage-admin '
old=$(lsof -tiTCP:8181 -sTCP:LISTEN 2>/dev/null || true)
if [ -n "$old" ]; then kill $old || true; sleep 2; fi
if lsof -tiTCP:8181 -sTCP:LISTEN >/dev/null 2>&1; then
  kill -9 $(lsof -tiTCP:8181 -sTCP:LISTEN) || true
  sleep 1
fi
cd /home/bupt/manage_deploy/backend
nohup /home/bupt/miniconda3/bin/python3.13 -m uvicorn main:app --host 0.0.0.0 --port 8181 \
  >/tmp/manage_deploy_backend.log 2>&1 < /dev/null &
echo $! > /home/bupt/backend.pid
sleep 2
tail -30 /tmp/manage_deploy_backend.log
'

# 4. 健康检查。
curl -sS http://10.112.244.94:8181/api/nodes | python3 -m json.tool >/dev/null
curl -sS http://10.112.244.94:8182/ | head
```

如果需要跑后端完整测试，优先使用：

```bash
cd /Users/yanjia/codes/manage_deploy/backend
PYTHONPATH=. ./venv/bin/python -m pytest tests/test_conversations_api.py tests/test_business_tasks_api.py tests/test_port_plan.py -q
```

如果只改文档，不需要重启部署；如果只改后端脚本但脚本要在管理节点执行，需要 rsync；如果只改前端，必须 `npm --prefix frontend run build` 后再 rsync。

| 场景 | 固定做法 |
| --- | --- |
| 本地开发目录 | `/Users/yanjia/codes/manage_deploy` |
| 管理节点部署目录 | `/home/bupt/manage_deploy` |
| 管理节点后端 Python | `/home/bupt/miniconda3/bin/python3.13` |
| 管理节点后端端口 | `8181` |
| 管理节点前端端口 | `8182`，静态文件来自 `/home/bupt/manage_deploy/frontend/dist` |
| 管理节点后端日志 | `/tmp/manage_deploy_backend.log` |
| 管理节点后端 PID | `/home/bupt/backend.pid` |
| 计算节点 Node Agent 运行方式 | 优先使用 `10.112.244.94:5000/node-agent:dev` 容器，host 网络，监听 `8001` |
| 终端节点 Node Agent 运行方式 | 优先使用 `10.112.244.94:5000/node-agent:dev` 容器，host 网络，监听 `8001` |
| Node Agent 源码同步目录 | `/opt/manage_deploy/node_agent`，仅用于源码留档或临时宿主机调试 |
| Node Agent 端口 | `8001` |
| Node Agent GPU 探测 | 容器内必须能访问 `nvidia-smi`；宿主调试时显式设置 `NVIDIA_SMI_BIN=/usr/bin/nvidia-smi` |
| 管理节点代码更新 | 本地验证后 `rsync` 到 `/home/bupt/manage_deploy`，不在远端 `git pull` |

固定执行顺序：

1. 本地确认分支和变更。
2. 本地跑必要测试和 `npm --prefix frontend run build`。
3. 用本文 rsync 命令同步到管理节点。
4. 用本文命令重启后端。
5. 如改了 `node_agent/`，同步并重启所有 Node Agent。
6. 如改了 worker 或 Dockerfile，重新构建并推送镜像。
7. 同步节点资源，跑用户端自启动 E2E 和批量业务测评回归。

禁止事项：

- 不在管理节点 `/home/bupt/manage_deploy` 里直接 `git pull`，除非明确重新初始化为标准 Git 部署目录。
- 不使用管理节点遗留的 `/home/bupt/manage_deploy/backend/venv`。
- 不在仓库根目录用系统 `python3` 直接跑后端脚本。
- 不让 Node Agent 从 `/tmp`、`~/tmp` 或一次性同步目录长期运行；正式运行优先用固定容器。
- 不临时换 Python 路径解决导入报错。缺依赖时先修固定环境。
- 不把“本地开发启动命令”和“管理节点生产启动命令”混用。本地可以用 `backend/venv`，管理节点只能用 `/home/bupt/miniconda3/bin/python3.13`。

## 3. Node Agent 标准化更新

正式运行优先更新并重启 Node Agent 容器。只有改了 `node_agent/` 时才需要重新构建镜像：

```bash
NODE_AGENT_IMAGE=10.112.244.94:5000/node-agent \
NODE_AGENT_TAG=dev \
NODE_AGENT_PUSH=1 \
NODE_AGENT_PLATFORM=linux/amd64 \
./scripts/build_node_agent.sh
```

每台被管理节点拉取并重启容器：

```bash
for host in manage-compute-1 manage-compute-2 manage-compute-3; do
  ssh "$host" '
    docker pull 10.112.244.94:5000/node-agent:dev
    docker rm -f manage-node-agent node-agent-compute-1 node-agent-compute-2 node-agent-compute-3 2>/dev/null || true
    docker run -d --name manage-node-agent --restart unless-stopped \
      --network host --privileged --gpus all \
      -e AGENT_PORT=8001 \
      -e DOCKER_SOCKET=unix:///var/run/docker.sock \
      -e NVIDIA_DRIVER_CAPABILITIES=compute,utility \
      -e NVIDIA_SMI_BIN=/usr/bin/nvidia-smi \
      -v /var/run/docker.sock:/var/run/docker.sock \
      10.112.244.94:5000/node-agent:dev
    sleep 2
    curl -sS http://127.0.0.1:8001/health
    echo
    curl -sS http://127.0.0.1:8001/resources
    echo
  '
done
```

如果 `/resources` 中 GPU 仍为 0，但宿主机 `/usr/bin/nvidia-smi -L` 能看到显卡，优先检查 Agent 容器是否用 `--gpus all` 启动，并确认容器内 `nvidia-smi -L` 能输出显卡。仅挂载 `/usr/bin/nvidia-smi` 不够，容器还需要 NVIDIA runtime 注入驱动库，例如 `libnvidia-ml.so`。这只影响节点资源展示，不等价于业务容器不能使用 GPU；是否影响业务要以 worker 容器日志和业务指标为准。

### 计算节点驱动与防火墙注意事项

当前 compute-1/2/3 均按 580 系列 NVIDIA 驱动验证：

| 节点 | 驱动 | GPU 画像 |
| --- | --- | --- |
| compute-1 | `580.159.03` | `1 × NVIDIA TITAN Xp` |
| compute-2 | `580.105.08` | `1 × NVIDIA TITAN X (Pascal)` |
| compute-3 | `580.126.09` | `8 × Tesla P40` |

矩阵计算镜像维持 `nvidia/cuda:12.4.1-runtime-ubuntu22.04` 和 `cupy-cuda12x`，不再为 compute-1 单独降级镜像。若驱动升级后外部无法连接 SSH/Node Agent，但本机访问 `10.112.*` 地址正常，先检查 `nftables` 规则集：

```bash
sudo tcpdump -ni enp6s0 'tcp port 2345 or tcp port 8001'
sudo nft list ruleset
```

如果抓包能看到外部 SYN 到达但本机没有回包，临时恢复命令为：

```bash
sudo nft flush ruleset
```

确认恢复后，再按现场安全要求补最小放行规则；不要反复重启机器或修改 `sshd_config`。`ssh.socket inactive` 不代表 SSH 异常，只要 `ssh.service` 运行且 `ss -lntp` 显示 `0.0.0.0:2345` 即可。

本地 `node_agent/` 源码也可以同步到每台被管理节点的固定目录，作为运维留档或临时宿主机调试：

```bash

  rsync -az --delete /Users/yanjia/codes/manage_deploy/node_agent/ "$host":/tmp/node_agent_sync/
  ssh "$host" 'sudo rsync -az --delete /tmp/node_agent_sync/ /opt/manage_deploy/node_agent/'

```

临时宿主机方式重启 Node Agent：

```bash
for host in manage-compute-1 manage-compute-2 manage-compute-3; do
  ssh "$host" '
    old=$(pgrep -f "uvicorn main:app --host 0.0.0.0 --port 8001" || true)
    if [ -n "$old" ]; then sudo kill $old || kill $old || true; sleep 1; fi
    cd /opt/manage_deploy/node_agent
    NVIDIA_SMI_BIN=/usr/bin/nvidia-smi \
    PYTHONPATH=/opt/manage_deploy/node_agent \
    nohup /usr/local/bin/python3.11 -m uvicorn main:app --host 0.0.0.0 --port 8001 \
      >/var/log/manage_deploy/node_agent.log 2>&1 < /dev/null &
    sleep 2
    curl -sS http://127.0.0.1:8001/health
    echo
    curl -sS http://127.0.0.1:8001/resources
    echo
  '
done
```

如果某台机器 `/usr/local/bin/python3.11` 不可用，应先固定安装路径或创建软链接，不要临时换成系统 Python 3.6/3.10 混跑。

Node Agent 更新后的验收标准：

- `curl http://127.0.0.1:8001/health` 返回健康。
- `curl http://127.0.0.1:8001/resources` 能返回 CPU、内存信息。
- compute 节点能返回 GPU 数量和型号。若显示 `gpu_count=0`，先检查 `NVIDIA_SMI_BIN` 和 Agent 启动用户环境。
- `pgrep -af "uvicorn main:app.*8001"` 中运行目录应对应 `/opt/manage_deploy/node_agent`，不能再指向 `/tmp`。

## 9. 环境路径问题排查清单

每次遇到“本地能跑、部署后不生效、GPU 识别不到、导入依赖失败、页面还是旧版”时，按以下顺序查，不要先猜业务代码问题。

### 管理节点后端

```bash
ssh manage-admin '
echo "PWD check"
readlink -f /home/bupt/manage_deploy/backend
echo "python check"
/home/bupt/miniconda3/bin/python3.13 -c "import sys; print(sys.executable)"
echo "process check"
pgrep -af "python3.13 -m uvicorn main:app.*--port 8181" || true
echo "port check"
lsof -nP -iTCP:8181 -sTCP:LISTEN || true
echo "log tail"
tail -40 /tmp/manage_deploy_backend.log
'
```

正确现象：

- `sys.executable` 是 `/home/bupt/miniconda3/bin/python3.13`。
- 监听 `8181` 的进程命令包含 `python3.13 -m uvicorn main:app`。
- 日志来自 `/tmp/manage_deploy_backend.log`。

错误现象和处理：

| 现象 | 处理 |
| --- | --- |
| 进程使用 `backend/venv/bin/python` | 停掉旧进程，用本文后端重启命令启动 |
| 进程使用系统 `python3` | 停掉旧进程，用 `/home/bupt/miniconda3/bin/python3.13` |
| 日志里 `ModuleNotFoundError` | 先确认 Python 路径；不要临时 `pip install` 到错误环境 |
| 页面还是旧版 | 重新 `npm --prefix frontend run build` 并 rsync `frontend/dist` |
| 前端更新后旧页面左侧菜单点击没反应 | 通常是浏览器仍运行旧入口 JS，而旧 hash chunk 已被 rsync 更新删除。当前前端已在路由层自动检测并刷新重试；若用户卡在更新前打开的老页面，手动刷新一次即可。 |

### Node Agent

```bash
for host in manage-compute-1 manage-compute-2 manage-compute-3; do
  echo "==== $host"
  ssh "$host" '
    echo "process"
    pgrep -af "uvicorn main:app.*--port 8001" || true
    echo "working dir"
    readlink -f /opt/manage_deploy/node_agent || true
    echo "health"
    curl -sS http://127.0.0.1:8001/health || true
    echo
    echo "resources"
    curl -sS http://127.0.0.1:8001/resources || true
    echo
    echo "nvidia-smi"
    /usr/bin/nvidia-smi -L 2>/dev/null || nvidia-smi -L 2>/dev/null || true
  '
done
```

正确现象：

- Node Agent 文件目录是 `/opt/manage_deploy/node_agent`。
- 监听 `8001` 的进程不是从 `/tmp` 目录启动。
- compute 节点 `/resources` 能看到 GPU 数量和型号。

错误现象和处理：

| 现象 | 处理 |
| --- | --- |
| 进程命令或日志指向 `/tmp` | 按“Node Agent 标准化更新”重新同步并启动 |
| `/resources` GPU 为 0 但 `nvidia-smi -L` 有显卡 | 检查启动命令是否带 `NVIDIA_SMI_BIN=/usr/bin/nvidia-smi` |
| `curl :8001/health` 不通 | 先查进程，再查防火墙和端口占用 |

### 脚本执行目录

脚本执行目录以前文“脚本执行目录约定”为准。排查时重点看三件事：

- 是否在正确目录执行：后端包脚本通常在 `backend` 目录；仓库级 E2E 脚本通常在仓库根目录。
- 是否用了正确 Python：本地用 `backend/venv`，管理节点用 `/home/bupt/miniconda3/bin/python3.13`。
- 是否显式设置 `PYTHONPATH`：需要导入后端包的脚本应设置 `PYTHONPATH=.` 或按脚本说明设置。

如果脚本需要从仓库根目录读取 `frontend/`、`workers/` 或 `docs/`，优先在脚本里使用绝对路径或显式参数，不要依赖当前 shell 的随机工作目录。
