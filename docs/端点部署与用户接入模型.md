# 端点部署与用户接入模型

本文记录 source / compute / sink 在测评模式和真实用户接入模式下的边界。逻辑 DAG 仍保持 `source -> compute -> sink`，但 DAG 角色不等于平台一定要部署容器；是否部署由工单的 `runtime_config.platform_deployment.deployable_roles` 决定。

## 两类运行模式

### 自动化测评模式

业务目标成功率测评使用运营商可控测试设备模拟业务源端和目的端。

- `platform_deployment.mode = automated_benchmark`
- `deployable_roles = ["source", "compute", "sink"]`
- 平台通过 Node Agent 部署三类容器。
- 平台为 source / compute / sink 分配容器端口并做 preflight。
- sink 汇总结果并上报业务指标。
- 用于 `/benchmark` 页面批量创建 30 个工单、运行测评和统计成功率。

当前代码中，自动化测评最稳定的识别方式是 `task_orders.is_benchmark=true` 或 `deployable_roles=["source","compute","sink"]`；`mode=automated_benchmark` 是设计口径，不要求每条历史或当前测评工单都显式写入。

这类 source/sink 是测试设备或仿真终端，不代表平台控制真实用户终端。它的价值是保证输入 profile、镜像、路由放置、指标采集和结果归档可复现。

### 真实用户接入演示模式

普通用户通过对话提交工单时，source/sink 默认是外部用户终端或外部系统。

- `platform_deployment.mode = user_access_demo`
- 默认 `deployable_roles = ["compute"]`
- 平台只部署运营商可控的 compute 容器。
- source 用户端主动访问平台返回的 compute 接入地址。
- compute 在无 sink 容器时直接上报业务指标，并把结果保留在 `/result` 供演示端查询。
- compute-only 容器会显式注入较长外部输入等待窗口，默认 `PEER_WAIT_TIMEOUT_SEC=3600`，避免用户端稍晚接入就导致任务失败。
- 如果目的端是运营商可控存储或汇聚节点，可将部署策略调整为 `["compute", "sink"]`。

真实用户终端端口不由平台分配。如果需要把结果推给外部目的端，应由用户或外部系统显式提供 `callback_url`；平台记录该地址、不参与端口冲突检查，compute 在无受控 sink 时会 best-effort POST 回调，回调失败只记录日志，不影响平台业务指标上报。

本文后续提到的 compute-only 均指 `platform_deployment.mode=user_access_demo` 且 `deployable_roles=["compute"]`，不是单独的第三种运行模式。

为方便演示，endpoint 镜像内提供了目的端接收器入口。用户或演示人员可在目的端设备手动启动 receiver：

```bash
# 矩阵计算目的端
docker run --rm --network host \
  10.112.244.94:5000/matmul-endpoint:dev \
  python /app/src/receiver_main.py --port 9000

# 视频推理目的端
docker run --rm --network host \
  10.112.244.94:5000/video-endpoint:dev \
  python /app/src/receiver_main.py --port 9000
```

receiver 提供 `POST /callback` 接收结果，浏览器打开 `http://<destination-business-ip>:9000/` 可查看最近工单结果；`GET /latest` 和 `GET /orders/{order_id}` 用于脚本查询。多个工单可共用同一端口，结果通过 `order_id` 区分。

## 角色职责

| 角色 | 逻辑含义 | 测评模式 | 用户接入模式 |
|------|----------|----------|--------------|
| source | 业务输入来源 | 平台部署测试 source 容器，生成或读取固定输入 | 用户终端自行启动应用或页面，主动提交输入 |
| compute | 核心计算或推理节点 | 平台部署，路由系统选择节点和 GPU | 平台部署，向用户端暴露接入端口 |
| sink | 结果汇总和归档 | 平台部署，接收结果并上报指标 | 默认不部署；可替换为运营商可控存储/汇聚节点 |

## 接口与数据模型

工单运行配置示例：

```json
{
  "platform_deployment": {
    "mode": "user_access_demo",
    "deployable_roles": ["compute"],
    "endpoint_policy": "source/sink are external user access endpoints by default",
    "external_endpoints": {
      "sink": {
        "callback_url": "https://user.example.com/api/result-callback"
      }
    }
  }
}
```

对话草稿支持通过 `PATCH /api/conversations/{id}/draft` 显式写入：

```json
{
  "callback_url": "https://user.example.com/api/result-callback"
}
```

确认意图后，平台会把该地址同步写入 `runtime_config.business_task.callback_url`、`runtime_config.platform_deployment.external_endpoints.sink.callback_url` 和 routing DAG 的 sink 节点；物化 compute 容器时同时注入 `CALLBACK_URL` / `SINK_CALLBACK_URL` 环境变量。

路由系统仍消费完整 DAG。source/sink 的真实拓扑端点写在 `routing_input_dag.nodes[].fixed_topology_node_id`；路由系统通常只需要回写 compute：

```json
{
  "placements": [
    {"task_node_id": "compute", "topology_node_id": "compute-3", "gpu_device": "0"}
  ]
}
```

平台接收路由结果后：

1. 保存路由回写的 placements。
2. 只为 `deployable_roles` 中的角色创建容器。
3. 基于逻辑 DAG 和已部署容器生成 `network_bindings`。
4. 对外部 source -> compute 链路返回 compute 的业务面 IP 和端口。
5. 对 compute -> 外部 sink 链路返回用户显式提供的 `dst_callback_url`，并复用 `dst_access_url` 展示回调地址。
6. compute 处理完输入后，先上报平台业务指标；若存在 `CALLBACK_URL`，再短超时 best-effort POST 外部回调。

## 端口与指标

端口分配规则：

- 平台只给自己部署的容器分配端口。
- 外部用户终端不分配平台端口，不做端口冲突检查。
- compute-only 模式下，用户端使用 `network_bindings` 中的 compute 访问地址提交输入。
- compute-only 模式下，受控 compute 默认等待外部 source 输入 3600 秒；可通过 `platform_deployment.external_input_wait_timeout_sec` 调整。

指标上报规则：

- 三容器测评模式：sink 上报。
- compute-only 用户接入模式：compute 上报。
- compute + operator sink 模式：运营商侧 sink 上报。
- 用户终端可补充上报体验指标，但不是平台业务目标成功率的必要证据。

## 演示顺序

1. 打开 `/benchmark`，说明这是可控测试域的业务目标成功率测评。
2. Step 1 展示节点 baseline。
3. Step 2 创建 30 个固定 profile 测评工单。
4. Step 3 路由系统回写 compute，平台在测评模式下补齐 source/sink 并小批次运行。
5. 打开单个工单详情，展示业务输入、路由放置、GPU、实际指标、baseline 和阈值。
6. Step 4 展示成功率，要求可评价任务数不少于 30 且成功率不低于 90%。
7. 切到 `/intent-chat`，说明真实用户工单默认只部署 compute，source/sink 是外部接入端点。

## 当前实现状态

- 用户对话确认工单默认写入 `platform_deployment.mode=user_access_demo` 和 `deployable_roles=["compute"]`。
- benchmark 工单继续使用 `deployable_roles=["source","compute","sink"]`。
- `network_bindings` 已支持从逻辑 DAG 补充外部端点到受控容器的链路。
- matmul/video compute 在没有 sink 时可直接上报指标。
- 工单详情页展示部署模式、受控部署角色、业务链路接入地址（`dst_access_url`）及外部用户接入说明。
- 工单详情页可展示外部 sink 回调地址（`dst_callback_url`），compute 容器会读取 `CALLBACK_URL` / `SINK_CALLBACK_URL` 并在无受控 sink 时尝试回调。
- endpoint 镜像提供 `receiver_main.py`，支持用户目的端手动接收回调并在浏览器查看最近结果。
- 提供 `scripts/user_source_client.py` 轻量用户端客户端与 `backend/scripts/e2e_compute_only_access.py` 验证脚本。
