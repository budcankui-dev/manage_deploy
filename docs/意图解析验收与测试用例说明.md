# 意图解析验收与测试用例说明

本文用于准备意图解析功能演示和“意图参数解析准确率”验收。它解释用户能输入什么、系统如何追问和校验、评测数据集怎么构成、页面结果怎么看，以及遇到测试方追问时建议如何回答。

## 1. 验收关注点

下周意图解析相关测试建议分成两类：

| 测试项 | 验证目标 | 推荐页面 |
|--------|----------|----------|
| 功能性演示 | 自然语言能解析为结构化任务参数，并在参数完整后提交工单 | 用户端 `/intent-chat` |
| 准确率用例 | 固定数据集逐条解析，计算字段级意图参数解析准确率 | 管理端 `/intent-evaluation` |

功能性演示不建议把 DAG JSON、内部路由细节作为主要验收点。演示重点是：系统能听懂业务需求、能识别缺失参数、能阻止非法提交、能生成工单。

## 2. 当前支持的任务类型

当前真实业务演示重点是两类：

| 任务类型 | 系统标识 | 必填业务参数 | 说明 |
|----------|----------|--------------|------|
| 矩阵乘法计算任务 | `high_throughput_matmul` | 源节点、目的节点、开始/结束时间、矩阵规模、批次数、路由策略 | 端到端演示和业务目标测评主链路之一 |
| 视频 AI 推理任务 | `low_latency_video_pipeline` | 源节点、目的节点、开始/结束时间、固定视频规格、抽检帧数、帧率、路由策略 | 使用镜像内置测试视频和 YOLOv5n 检测模型，可展示检测框、中文标签、推理时延和结果预览 |

意图准确率数据集还覆盖八类业务模态和扩展任务标签，用于证明解析器具备更宽的参数抽取能力。当前可部署闭环以矩阵计算和视频推理为主，其他扩展类型更适合作为意图解析样本，不建议在功能演示中承诺完整部署。

## 3. 多轮补全与追问

系统支持多轮补全。第一轮如果缺参数，解析结果会进入“参数不全”，页面会列出缺失项，确认提交按钮不会放行。用户继续输入补充信息后，后端会把新信息与上一版草稿合并。

推荐演示样例：

```text
视频AI推理任务，从 h1 到 h2，720p视频，100帧，现在开始跑2小时，低时延策略
```

预期结果：系统识别任务类型、源节点、目的节点、固定视频规格、抽检帧数和时间，但提示缺少帧率。

继续输入：

```text
补充帧率 30fps
```

预期结果：草稿补齐为“参数完整”，可以确认提交任务。

注意：多轮补全适合补缺失参数，不建议在同一个对话里反复改成完全不同的任务。一个对话生成一个工单后，页面会提示新建对话继续提交新任务。

## 4. 节点与非法值校验

源节点和目的节点必须是系统已登记的运营商端点别名、拓扑节点 ID 或业务面 IP。当前用户输入最建议使用别名：

| 节点类型 | 可用输入 |
|----------|----------|
| 运营商端点 | `h1` 至 `h13` |
| 计算节点 | `compute-1`、`compute-2`、`compute-3` 由系统或路由模块选择，不作为源/目的输入 |

如果输入不存在的节点，例如：

```text
矩阵乘法任务，从 unknown-node 到 h2，1024阶矩阵，50批，现在开始跑2小时，资源保障策略
```

预期结果：系统提示“源节点不存在”，草稿保持“参数不全”，不能提交工单。

业务参数也有范围校验：

| 参数 | 合法范围 |
|------|----------|
| 矩阵规模 | 128 至 32768 的整数 |
| 批次数 | 1 至 10000 的整数 |
| 视频抽检帧数 | 1 至 100000 的整数 |
| 视频帧率 fps | 1 至 240 的整数 |
| 视频规格/分辨率 | `480p`、`720p`、`1080p`、`2k`、`4k`、`8k`。当前验收镜像使用内置固定测试视频，分辨率和帧率用于标识本次测试 profile，不表示现场上传任意视频。 |

校验位置有两层：前端会在确认提交前提示，后端 `confirm-intent` 也会再次校验。因此即使绕过页面直接调接口，缺失或非法参数也不会生成工单。

## 5. 时间格式支持

验收演示建议使用相对时间表达，因为最稳定、最容易解释：

| 推荐表达 | 解析效果 |
|----------|----------|
| `现在开始跑2小时` | 开始时间为当前时间，结束时间为 2 小时后 |
| `立即运行60分钟` | 开始时间为当前时间，结束时间为 60 分钟后 |
| `马上执行半小时` | 开始时间为当前时间，结束时间为 30 分钟后 |
| `立即执行一小时半` | 开始时间为当前时间，结束时间为 90 分钟后 |

规则兜底解析器不把下面这类绝对时间作为推荐验收输入：

```text
2026-06-20 09:00开始，2026-06-20 11:00结束
```

如果启用在线大模型解析，系统可以处理更丰富的自然语言时间；但为了验收稳定性，建议测试用例统一使用“现在/立即/马上 + 持续时长”。

## 6. 推荐功能演示输入用例

矩阵计算完整用例：

```text
矩阵乘法任务，从 h1 到 h2，1024阶矩阵，50批，现在开始跑2小时，资源预留保障策略
```

视频推理完整用例：

```text
视频AI推理任务，从 h3 到 h4，720p视频，100帧，30fps，现在开始跑2小时，低时延转发策略
```

缺参追问用例：

```text
视频AI推理任务，从 h1 到 h2，720p视频，100帧，现在开始跑2小时，低时延策略
```

非法节点用例：

```text
矩阵乘法任务，从 ghost-node 到 h2，1024阶矩阵，50批，现在开始跑2小时
```

非法参数用例：

```text
视频AI推理任务，从 h1 到 h2，720p视频，100帧，300fps，现在开始跑2小时
```

上面非法参数可能先被解析成草稿，但确认提交时会提示 fps 需要在 1-240 之间。

## 7. 用户端演示交互怎么看

`/intent-chat` 页面建议按以下顺序讲：

1. 用户在聊天框输入自然语言业务需求。
2. 右侧“意图参数”展示任务类型、所属模态、源/目的节点、业务面 IP、时间、业务参数、路由策略。
3. 参数不完整时，中间会出现“参数待补充”卡片，并列出缺失项。
4. 参数完整后，页面显示“确认提交任务”按钮。
5. 高级提交选项中有“只生成节点分配方案，不启动任务”，默认不启用。
6. 对于用户端演示，右侧会给出源端和目的端容器启动命令；目的端端口按任务类型固定，避免用户手填端口导致冲突或解释复杂。

固定目的端端口：

| 任务类型 | 目的端接收端口 |
|----------|----------------|
| 矩阵乘法计算任务 | `9000` |
| 视频 AI 推理任务 | `9100` |

同一台目的端的同类任务可以共用一个接收程序，业务结果按统一工单 ID 区分。

## 8. 意图测评页面怎么看

`/intent-evaluation` 用于固定数据集评测。建议这样解释：

| 页面内容 | 含义 |
|----------|------|
| 评测流程卡片 | 展示当前评测进度、准确率和运行状态 |
| 数据集说明 | 说明样本覆盖的任务类型、模态、节点槽位和缺失/错误场景 |
| 单条检测 | 抽查一条自然语言输入，查看系统解析出的结构化参数 |
| 样本列表 | 展开后可查看原始样本 JSON、系统解析 labels JSON，以及“标签值 / 解析值 / 是否正确”的字段对照表 |

固定数据集当前规模为 360 条，覆盖：

| 维度 | 覆盖内容 |
|------|----------|
| 源/目的节点 | `h1-h13` 终端节点槽位 |
| 任务 | 矩阵、视频、LLM 文本生成和八类模态扩展任务 |
| 路由策略 | 成本开销保障、低时延转发、资源预留保障、完成时间优先、资源负载均衡 |
| 异常样本 | 缺少源节点、缺少目的节点、缺少时间、错误源节点、错误目的节点 |

“人工标注答案”不是人工现场填写，而是数据集预先定义的标准答案，用于自动评分。准确率是系统解析输出与标准答案逐字段比对后的结果。

### 数据集字段口径

数据集采用“用户输入 + 人工标注答案 + 评测辅助信息”的结构，避免把工程内部字段和人工标注混在一起。

| 字段 | 含义 | 是否展示给专家 |
|------|------|----------------|
| `utterance` | 用户自然语言输入 | 是 |
| `labels` | 人工标注答案，包括任务类型、模态、策略、源/目的节点、运行时间和业务参数 | 是，页面以中文卡片展示 |
| `evaluation` | 评测辅助信息，例如样本是否有效、预期缺失哪些字段 | 是，但只展示中文解释 |
| 技术 JSON | 页面排障用原始结构 | 默认折叠，不作为讲解重点 |

`labels` 中的枚举值使用中文，例如“视频AI推理任务”“低时延转发模态”“资源预留保障”。系统评分时会自动映射为内部稳定代码，因此既便于专家理解，也不会影响自动评测。

运行时间优先标注为相对时长，例如“当前时间开始，运行 60 分钟”。这样做是为了避免固定日期过期，也便于每次验收重复运行。系统实际解析后仍会生成明确的开始时间和结束时间。

业务目标指标，例如 `effective_gflops`、`frame_latency_p90_ms`、比较符和单位，属于系统根据任务类型推导的验收指标，不作为人工标注答案写入数据集主标签。这样可以避免测试方误解为用户必须在自然语言里说出指标字段。

## 9. 常见追问回答

**为什么功能演示只说矩阵和视频？**

因为这两类已经有业务容器和结果展示链路，适合证明“解析 -> 提交 -> 部署/演示 -> 结果证据”。准确率数据集覆盖更宽的模态，是为了验证参数抽取能力。

**为什么时间用相对表达？**

验收重点是参数解析和工单提交，不是自然语言日历系统。相对时间表达稳定、可复现，也更贴近“现在提交一个业务”的演示场景。

**为什么目的端口不让用户手动填？**

为了演示清晰和避免同类型任务端口冲突，当前按任务类型固定端口。真实接入时也可以扩展为用户登记回调地址，但验收演示先保持简单。

**如果节点不存在怎么办？**

系统会提示节点不可用，草稿保持参数不全，不允许提交。路由系统只接收已登记拓扑节点或可解析的数据面 IP。

**如果 token 过期怎么办？**

页面请求返回 401 时会提示“登录已过期，请重新登录”，并跳转登录页。重新登录后不应停留在蒙板加载状态。
