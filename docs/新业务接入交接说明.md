# 新业务接入交接说明

本文是给其他模块同学的入口文档。目标是让业务提供方先把任务边界、参数、镜像和验收结果讲清楚，再进入系统适配，避免直接改代码时把意图解析、DAG、路由和容器参数混在一起。

## 先读什么

请按顺序阅读：

1. [新业务接入参数确认清单](./新业务接入参数确认清单.md)：业务提供方需要填写的参数、镜像、端口、指标和自然语言样例。
2. [意图参数数据集与评测页面维护说明](./意图参数数据集与评测页面维护说明.md)：本系统维护者扩展意图数据集和评测页面时必须遵守的格式。
3. [端点部署与用户接入模型](./端点部署与用户接入模型.md)：理解自动测评模式和真实用户接入模式的区别。

业务提供方不需要先理解本系统全部实现，只需要按清单交付可验证信息。

## 对方需要交付什么

最小交付包包括：

| 类型 | 内容 | 用途 |
|------|------|------|
| 业务说明 | 中文名称、英文标识、所属模态、推荐路由策略 | 前端展示、路由策略选择、数据集标签 |
| 用户参数 | 源节点、目的节点、运行时长、业务参数、合法范围 | 意图解析、多轮补全和提交校验 |
| 容器角色 | source / compute / sink 是否需要容器、镜像、端口、启动命令 | DAG 物化和用户端手动启动说明 |
| 资源建议 | GPU/CPU/内存、端口、健康检查、环境变量 | 调度部署和节点能力展示 |
| 成功证据 | 指标名、指标方向、单位、页面预览、结果上报方式 | 业务目标成功率和工单详情展示 |
| 自然语言样例 | 5-10 条标准、口语、缺参和错误节点样例 | 构建和扩充意图评测数据集 |

建议先只接一条最小闭环：能解析、能提交、能部署或手动启动、能展示结果。不要一开始就要求覆盖所有异常场景。

## 我们收到后改哪里

本系统维护者按以下顺序适配：

1. 扩展后端意图解析器，输出新的任务类型、模态、策略和业务参数。
2. 扩展 `backend/scripts/generate_intent_dataset.py`，加入新任务自然语言模板、`data_profile` 字段和 `labels`；重新生成 `datasets/intent_eval/multi_business.jsonl`，并运行 `backend/tests/test_generate_intent_dataset.py`。
3. 扩展前端展示字典，确保任务类型、模态、策略和业务参数都用中文展示。
4. 注册或重建任务模板，明确 source / compute / sink 角色、端口和环境变量。
5. 补业务 worker 或接入外部镜像，确认容器之间只通过网络通信传递业务数据。
6. 补测试：意图数据集生成、解析评分、工单创建、路由 payload、部署或手动演示。
7. 更新路由对接文档中该任务需要传给路由的模态、策略、端口、业务面 IP 和拓扑节点 ID。

## 验收标准

接入完成后至少满足：

- 用户端自然语言输入能解析出任务类型、源/目的节点、业务参数、运行时间和路由策略。
- 参数缺失或非法时不能提交工单，页面给出中文提示。
- 数据集评测页面能展示该任务样本，字段对照表包含“英文字段 / 中文含义 / 标签值 / 解析值 / 是否正确”。
- 工单能生成路由 DAG，DAG 中源/目的节点包含拓扑节点 ID 和数据面 IP，compute 节点包含可部署角色和资源需求。
- 自动测评模式可以由平台启动所需容器；用户接入模式可以给出用户端手动启动命令。
- 结果页面能展示专家能理解的业务结果，而不是只展示内部 JSON。

## 不要做什么

- 不把内部部署参数、镜像路径、容器环境变量写进意图数据集 `labels`。
- 不把一种模态写死成一种任务；一个模态可以覆盖多个任务。
- 不让专家页面展示 mock、规则兜底、内部 parser 名称或多个互相竞争的准确率。
- 不把 `compute-1~3` 当作用户源/目的槽位；用户源/目的当前使用 `h1-h13` 或可解析的数据面 IP。
- 不要求业务提供方理解本系统数据库结构后才能交付参数；数据库字段由系统维护者映射。
