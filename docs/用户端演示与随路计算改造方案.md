# 用户端演示与随路计算改造方案

本文记录当前已确认的用户端演示、自动化测评、数据集改造、前后端解析改造和在线意图测评验证要求。目标是让系统同时满足两类场景：

- 用户功能演示：源端和目的端设备由用户自行控制，平台不登录、不部署、不分配用户端口。
- 业务目标成功率测评：使用运营商管控测试端点，由平台一键自动部署 source / compute / sink，保证批量测评可复现。

本文是改造设计与后续任务清单，不表示所有条目已经实现。实现时应先保持现有自动化测评链路稳定，再逐步改造用户演示链路。

## 0. 当前实现状态

截至当前版本，已完成以下基础能力：

- 后端可解析用户填写的端点别名、拓扑节点 ID 或业务面 IP，并写入 source/sink DAG 节点。
- 用户演示模式默认 `deployable_roles=["compute"]`，平台只部署 compute；source/sink 作为外部用户端点。
- 高级模式支持 `route_only`，即只保存路由系统返回的 compute placement 和 metadata，不物化实例、不分配容器端口。
- 工单详情接口返回 `routing_decision` 摘要，聚合路由 placement、metadata、节点资源和 `node_baselines`，用于展示“本策略下推荐节点”。
- 用户端对话页支持保存源端、目的端和“仅生成节点分配方案”高级选项；目的端口按任务类型固定，不在主流程中要求用户手填。
- endpoint 镜像已提供 `receiver_main.py` 手动启动入口。目的端用户设备的浏览器访问地址就是 `ip:端口` 首页；后台 compute 通过同一容器的内部接口 `/callback` 推送结果，用户不需要知道这个路径；`/latest` 和 `/orders/{order_id}` 保留 JSON 查询接口。

仍待继续完善：

- 在线意图测评全量 360 条样本的稳定调用验证。
- 数据集补充更多“业务面 IP + 固定接收端口 + route-only”的自然语言样本。

## 1. 核心口径

### 1.1 两套运行模式

| 模式 | 源端 A | 算力节点 B | 目的端 C | 平台控制范围 | 主要用途 |
|------|--------|------------|----------|--------------|----------|
| 用户演示模式 | 用户自行控制的源端设备 | 平台部署 compute | 用户自行控制的目的端设备 | 只部署 compute | 展示真实用户预约、随路计算和结果送达 |
| 自动化测评模式 | 平台自动部署 source | 平台部署 compute | 平台自动部署 sink | 部署 source / compute / sink | 批量验证业务目标成功率 |

用户演示模式下，A/C 都是用户控制的终端设备。平台只接收用户登记的源端、目的端数据面信息，并把它们写入 DAG 和路由请求。平台不自动登录用户终端，不替用户分配本地端口。

自动化测评模式下，A/C 是运营商管控测试端点。平台可以自动启动 endpoint 镜像，是为了验收测评的自动化和可复现，不代表生产环境要控制真实用户终端。

### 1.2 随路计算解释

随路计算仍保持：

```text
A(source) -> B(compute) -> C(sink/receiver)
```

区别在于不同模式下 A/C 的控制方式不同：

- 用户演示模式：A/C 由用户手动启动演示程序，平台只部署 B。
- 测评模式：A/B/C 都由平台自动部署，用于业务目标成功率测评。

## 2. 用户演示模式设计

### 2.1 用户前端输入

用户端前端应支持输入或解析以下字段：

| 字段 | 是否必填 | 说明 |
|------|----------|------|
| 业务类型 | 是 | 当前支持矩阵计算和视频 AI 推理 |
| 源端数据面 IP 或节点别名 | 是 | 例如 `10.112.126.124` 或 `h1` |
| 目的端数据面 IP 或节点别名 | 是 | 例如 `10.112.20.40` 或 `h3` |
| 目的端接收端口 | 自动确定 | 按任务类型固定：矩阵任务 `9000`，视频任务 `9100` |
| 路由策略 | 否 | 默认使用资源预留保障，可选低时延转发等策略 |
| 业务参数 | 否 | 矩阵规模、批次数、视频样本等；可提供默认值 |

后端按任务类型把目的端数据面 IP 和固定端口拼成两类地址：用户界面只展示容器首页地址，后台注入 compute 的 `CALLBACK_URL` 使用内部 POST 地址。

```text
用户可见地址：http://<destination_business_ip>:<destination_port>
内部回调地址：http://<destination_business_ip>:<destination_port>/callback
```

注意：用户输入的是数据面地址，不是管理面地址。

### 2.2 节点解析规则

前端允许用户输入节点别名、拓扑节点 ID 或数据面 IP，但后端必须解析成完整的拓扑信息。解析不到则不允许提交。

| 用户输入 | 后端匹配 | 输出 | 是否合法 |
|----------|----------|------|----------|
| `h1` | 节点别名/hostname | `topology_node_id + business_ip` | 合法 |
| `h18001001` | 拓扑节点 ID/主机 ID | `topology_node_id + business_ip` | 合法 |
| `10.112.126.124` | `business_ip` 或 `business_ipv6` | `topology_node_id + business_ip` | 合法 |
| 未登记 IP | 无 | 无 | 不合法 |
| 管理面 IP | 原则上不作为业务端点 | 无或提示修正 | 不建议 |

生成 DAG 和传给路由系统时，应同时包含拓扑节点 ID 和数据面 IP：

```json
{
  "task_node_id": "source",
  "task_role": "source",
  "node_type": "external_user_endpoint",
  "deployable": false,
  "topology_node_id": "h18001001",
  "topology_alias": "h1",
  "business_ip": "10.112.126.124",
  "business_port": null
}
```

```json
{
  "task_node_id": "sink",
  "task_role": "sink",
  "node_type": "external_user_endpoint",
  "deployable": false,
  "topology_node_id": "h18005003",
  "topology_alias": "h3",
  "business_ip": "10.112.20.40",
  "business_port": 9000,
  "callback_url": "http://10.112.20.40:9000/callback"
}
```

路由系统优先使用 `topology_node_id` 做拓扑识别和算路，使用 `business_ip` / `business_port` 标识业务流。

### 2.3 手动启动用户端 endpoint

当前项目已拆分 endpoint Dockerfile：

- `workers/high-throughput-matmul/Dockerfile.endpoint`
- `workers/low-latency-video/Dockerfile.endpoint`

建议不再拆成“测评版”和“用户演示版”两个终端镜像，而是在同一个 endpoint 镜像里支持不同启动模式：

| 启动模式 | 启动方 | 用途 |
|----------|--------|------|
| `source` | 平台自动启动 | 自动化测评生成输入 |
| `sink` | 平台自动启动 | 自动化测评接收结果并上报指标 |
| `receiver` | 用户手动启动 | 用户演示模式下接收 compute 推送结果 |
| `manual-source` | 用户手动启动，可后续补充 | 用户演示模式下主动向 compute 提交输入 |

第一阶段建议优先实现目的端 `receiver`，源端输入可以先由前端或轻量脚本提交到 compute。这样演示步骤更少。

目的端手动启动示例。矩阵计算使用 `9000`。`ENDPOINT_*` 环境变量只用于页面展示和排查，不参与路由计算；真实路由仍以工单 DAG 和 `/result` 返回的 `network_bindings` 为准。

```bash
docker run --rm --network host \
  -e ENDPOINT_NODE_ALIAS=h2 \
  -e ENDPOINT_TOPOLOGY_NODE_ID=h18015002 \
  -e ENDPOINT_BUSINESS_IP=10.112.253.42 \
  -e ENDPOINT_BUSINESS_IPV6=2001:da8:215:6a01:xxxx:xxxx:xxxx:xxxx \
  10.112.244.94:5000/scientific-matmul-endpoint:dev \
  python /app/src/receiver_main.py --port 9000
```

视频 AI 推理使用 `9100`：

```bash
docker run --rm --network host \
  -e ENDPOINT_NODE_ALIAS=h2 \
  -e ENDPOINT_TOPOLOGY_NODE_ID=h18015002 \
  -e ENDPOINT_BUSINESS_IP=10.112.253.42 \
  -e ENDPOINT_BUSINESS_IPV6=2001:da8:215:6a01:xxxx:xxxx:xxxx:xxxx \
  10.112.244.94:5000/low-latency-video-endpoint:dev \
  python /app/src/receiver_main.py --port 9100
```

receiver 已提供：

| 接口 | 作用 |
|------|------|
| `POST /callback` | 接收 compute 推送的业务结果，属于内部接口路径，不在用户页面展示 |
| `GET /` | 浏览器展示最近任务结果、关键指标和视频带框预览图 |
| `GET /latest` | 返回最近任务结果 JSON |
| `GET /orders/{order_id}` | 按工单 ID 查看单个任务结果 |

多实例不要靠不同端口区分。所有任务可以共用同一个固定端口，结果体里必须带 `order_id`。浏览器打开 `GET /` 时默认展示最近一次结果，页面顶部“切换已接收工单”下拉可以按工单 ID 切换历史结果；`GET /latest` 和 `GET /orders/{order_id}` 用于 JSON 排查。

正式验收/实测时必须记住：业务数据流走数据面 IPv6。`10.112` IPv4 当前主要用于管理面、SSH、Node Agent 和阶段性开发演示；后续相关网络切换后，应更新 `nodes.business_ipv6`，将后端 `PREFER_BUSINESS_IPV6=true`，并确认 `network_bindings`、`PEER_*_URL` 和 receiver 页面展示的数据面 IPv6 一致。不要把浏览器访问页面用到的管理/过渡 IPv4 误认为最终业务流地址。

### 2.4 特殊模式：只路由不部署

除默认用户演示模式外，系统需要支持一个非默认高级选项：只解析意图、生成工单、调用路由系统计算 A -> B -> C 路径和 compute 节点，但不自动部署任何容器。

适用场景：

- 用户只想验证自然语言能否正确解析出业务参数、源端、目的端和路由策略。
- 用户或其他业务模块希望自行手动启动 source / compute / receiver，只借用平台的意图解析和路由推荐能力。
- 演示时需要展示“系统给出的算力节点选择”，但暂时不希望平台调度部署。

该模式不应作为默认入口，建议放在高级设置中，名称可为：

```text
仅生成路由方案，不自动部署
```

运行配置建议：

```json
{
  "platform_deployment": {
    "mode": "route_only",
    "deployable_roles": [],
    "auto_deploy": false
  }
}
```

只路由不部署模式下：

- 平台仍创建工单、生成 DAG、调用路由系统。
- 路由系统仍回写 compute placement、路径、成本、租金、解释等 metadata。
- 平台不物化 TaskInstance，不调用 Node Agent，不分配容器端口。
- 工单状态展示为“已生成路由方案”或“待用户手动启动”。
- 页面展示推荐 compute 节点、路径、源/目的数据面 IP、路由策略和算法解释。
- 业务目标成功率统计不纳入 route-only 工单。

该模式需要避免和默认用户演示模式混淆。默认用户演示模式仍应由平台自动部署 compute；route-only 只是特殊调试/演示能力。

## 3. 当前两个业务的适配

### 3.1 矩阵计算

用户演示模式：

```text
用户填写源端数据面 IP、目的端数据面 IP、目的端端口和矩阵参数
-> 平台解析意图并生成 DAG
-> 路由系统选择 compute 节点
-> 平台只部署 compute
-> 用户源端/前端/脚本向 compute 提交矩阵任务
-> compute 完成计算
-> compute 上报平台指标
-> compute POST 到目的端 receiver
-> 平台工单详情和目的端 receiver 页面均可查看结果
```

目的端 receiver 展示内容：

- `order_id`
- 矩阵规模
- 批次数
- 计算耗时
- GFLOPS 或业务指标
- 结果摘要
- compute 节点和 GPU 信息

自动化测评模式保持三容器链路：

```text
source -> compute -> sink
```

sink 继续上报业务目标指标，用于成功率统计。

### 3.2 视频 AI 推理

用户演示模式：

```text
用户填写源端数据面 IP、目的端数据面 IP、目的端端口和视频任务参数
-> 平台解析意图并生成 DAG
-> 路由系统选择 compute 节点
-> 平台只部署 YOLO compute
-> 用户源端/前端/脚本提交视频任务或固定视频样本
-> compute 完成推理
-> compute 上报平台指标
-> compute POST 到目的端 receiver
-> 平台工单详情和目的端 receiver 页面均可查看带框结果
```

第一阶段不做实时视频流推流，先推送结果 JSON、实时进度事件和带框图片。compute 推理过程中可以向 receiver 发送 `event_type=progress` 的进度回调，receiver 页面自动刷新展示增长中的抽帧样本；推理完成后再发送 `event_type=final` 的最终回调。业务目标成功率、平台指标和正式验收判定只使用 final 结果，progress 只作为页面运行证据。receiver 展示内容：

- `order_id`
- 检测模型
- 原始测试视频或视频样本
- 帧序号
- 中文分类标签
- 检测框坐标
- 带框预览图
- 抽帧检测证据列表
- 推理时延趋势
- 单帧时延和 P90 时延
- compute 节点和 GPU 信息

自动化测评模式保持固定测试视频和三容器链路：

```text
source -> compute -> sink
```

sink 继续上报 `frame_latency_p90_ms`、检测框和带框预览图，用于业务目标成功率统计。

## 4. 路由对接影响

### 4.1 用户演示模式

用户演示模式下，路由系统仍可看到完整 DAG，但 source/sink 是外部用户端点：

- `source.deployable=false`
- `sink.deployable=false`
- `compute.deployable=true`
- source/sink 需要同时携带 `topology_node_id` 和 `business_ip`
- sink 额外携带 `business_port` 和 `callback_url`

路由系统主要回写 compute 放置结果：

```json
{
  "placements": [
    {
      "task_node_id": "compute",
      "topology_node_id": "compute-3",
      "gpu_device": "0"
    }
  ],
  "metadata": {
    "path": ["h18001001", "compute-3", "h18005003"]
  }
}
```

平台接收路由结果后只物化 compute 容器，并将目的端回调信息注入 compute。当前直接注入的环境变量包括：

- `CALLBACK_URL`
- `SINK_CALLBACK_URL`
- `ORDER_ID`
- `TASK_TYPE`
- `ROUTING_STRATEGY`
- `MODALITY`
- `TASK_PRIORITY`
- `BUSINESS_TASK_JSON`
- `PLATFORM_DEPLOYMENT`
- `ROUTING_RESULT`

源端/目的端业务面 IP、目的端口、DAG 节点信息和路由结果会放在 `BUSINESS_TASK_JSON`、`PLATFORM_DEPLOYMENT` 或 `ROUTING_RESULT` JSON 中。worker 如果需要这些字段，应从 JSON 中读取，不要假设存在 `SOURCE_BUSINESS_IP`、`DESTINATION_BUSINESS_IP`、`DESTINATION_BUSINESS_PORT` 这类同名环境变量。

### 4.2 自动化测评模式

测评模式仍为：

- `source.deployable=true`
- `compute.deployable=true`
- `sink.deployable=true`
- 平台为三类容器自动分配端口
- 路由和部署仍沿用当前成功率测评闭环

测评模式不依赖用户手动启动 receiver，避免用户终端环境影响业务目标成功率。

### 4.3 路由接口是否需要改

核心 HTTP 接口建议保持不变：

| 接口 | 是否变化 | 说明 |
|------|----------|------|
| `GET /api/routing-orders` | 不变 | 仍返回待路由工单和 `routing_input_dag` |
| `PATCH /api/routing-orders/{order_id}/claim` | 不变 | 仍用于并发领取工单 |
| `POST /api/routing-orders/{order_id}/result` | 不变 | 仍回写 compute placement；平台随后物化 compute 并返回 `network_bindings` |
| `POST /api/routing-orders/{order_id}/network-ready` | 不变 | 路由系统按 `network_bindings` 下发流表/QoS 后调用 |
| `GET/POST /api/routing-resource-events` | 不变 | 仍用于 GPU 等平台资源释放确认 |

因此，路由对接不建议新增一套接口。改造重点是 DAG 字段和处理时序，而不是换接口。

### 4.4 路由系统需要适配的点

用户演示模式下，路由系统需要按下面规则处理：

1. 读取 source/sink 的 `topology_node_id` 和 `business_ip`。这两个字段必须同时存在；`business_ip` 是数据面 IP，不是管理面 IP。
2. 识别 `deployable=false`：source/sink 是用户手动控制的端点，不需要路由系统回写 source/sink placement。
3. 仍然基于完整 A -> B -> C DAG 计算路径，但回写给平台的 placement 通常只包含 compute。
4. compute 的动态业务端口在平台物化实例后才知道，因此路由系统不能只靠输入 DAG 提前下发完整业务流规则。
5. `/result` 返回的 `network_bindings` 是最终下发流表/QoS 的依据，里面包含实际 compute 业务面 IP/端口，以及目的端用户登记的业务面 IP/端口。
6. 用户目的端 receiver 是否已启动，不属于路由资源分配成功的判断。`network-ready` 表示网络路径或 QoS 已配置，不表示用户应用一定已经接收成功。
7. 用户端回调失败不应该触发路由系统释放或回滚资源；平台会保留结果并记录回调失败状态。

用户演示模式的推荐时序：

```text
路由系统读取 pending 工单
 -> claim 工单
 -> 解析 DAG：source/sink 是外部用户端点，compute 是平台部署节点
 -> 根据 source.topology_node_id、sink.topology_node_id、business_ip 和资源需求选择 compute
 -> POST /result，仅回写 compute placement 和算法 metadata
 -> 平台物化 compute，分配 compute 业务端口，返回 network_bindings
 -> 路由系统根据 network_bindings 下发 A->B、B->C 流表/QoS
 -> POST /network-ready
 -> 平台启动 compute 或注册启动计划
```

自动化测评模式下，路由系统处理方式基本不变：

- source / compute / sink 都是平台可部署角色。
- 平台会为 source / compute / sink 自动分配端口。
- 路由系统仍主要选择 compute 节点和 GPU。
- `/result` 后仍以 `network_bindings` 作为最终业务流端口依据。

### 4.5 需要更新的路由对接文档点

当前 `docs/routing-system-integration-guide.md` 仍偏向“平台会在 source/sink 固定拓扑节点部署端点容器”的旧口径。实现本改造后，需要同步更新该对接文档：

1. 在“端点部署”说明中区分用户演示模式和自动化测评模式。
2. 明确用户演示模式下 `source/sink.deployable=false`，路由系统不要回写 source/sink placement。
3. 明确 DAG 中 source/sink 必须包含 `topology_node_id` 和数据面 `business_ip`，sink 还包含 `business_port`。
4. 明确 `network_bindings` 是路由系统下发真实业务流规则的最终依据，因为 compute 动态端口只有 `/result` 后才产生。
5. 明确 `network-ready` 只表示路由/网络规则就绪，不表示用户 receiver 应用已经成功接收结果。
6. 保持回写接口和资源释放接口不变，减少路由同学改造成本。

### 4.6 路由选择理由与节点好坏展示

验收演示时，用户端和管理端都应能看懂“为什么选择这个 compute 节点”。尤其当用户选择成本开销保障、低时延转发、资源预留保障、完成时间优先或资源负载均衡时，页面不能只显示一个节点名，还要展示该节点相对其他候选节点的指标依据。

建议展示为“路由决策说明”卡片，位置包括：

- 用户端工单详情。
- 管理端任务工单详情。
- 业务测评测试工单列表展开详情。
- 只路由不部署模式的路由方案页。

卡片内容建议：

| 展示项 | 来源 | 说明 |
|--------|------|------|
| 选中算力节点 | 路由回写 placement | 例如 `compute-3` |
| GPU 编号 | 路由回写 placement | 例如 `gpu_device=0` |
| 节点基线指标 | `node_baselines` 或路由 metadata | 矩阵显示 GFLOPS，视频显示 P90 时延 |
| 资源能力 | `nodes` / 资源同步结果 | GPU 型号、GPU 数量、CPU、内存等 |
| 策略匹配解释 | 路由 metadata | 说明本次为何选中该节点 |
| 候选节点对比 | 路由 metadata 或平台查询 baseline | 展示候选节点排序和关键指标 |
| 预计成本/租金 | 路由 metadata | 成本优先策略下重点展示 |
| 路径时延/跳数 | 路由 metadata | 低时延转发策略下重点展示 |
| 资源余量 | 路由 metadata | 资源预留保障、负载均衡策略下重点展示 |

不同策略的展示重点：

| 路由策略 | UI 重点 |
|----------|---------|
| 成本开销保障 | 展示预计成本、租金、成本排序，以及选中节点是否满足资源要求 |
| 低时延转发 | 展示源端 -> compute -> 目的端路径、估计链路时延、跳数或路径评分 |
| 资源预留保障 | 展示 GPU 独占、资源余量、是否满足 CPU/内存/GPU 需求 |
| 完成时间优先 | 展示节点 baseline、预计完成时间、计算能力排序 |
| 资源负载均衡 | 展示当前负载、已占 GPU、候选节点分散情况 |

平台侧可先用已有 `nodes` 和 `node_baselines` 做基础展示；路由系统可在 `/result` 的 `metadata` 字典里补充更丰富的算法解释，例如：

```json
{
  "metadata": {
    "algorithm": "latency-aware-v1",
    "selected_reason": "compute-3 has the lowest A->B->C path latency among available GPU nodes",
    "candidate_scores": [
      {
        "topology_node_id": "compute-3",
        "score": 0.92,
        "baseline": {"effective_gflops": 8120, "frame_latency_p90_ms": 9.3},
        "estimated_path_latency_ms": 12.4,
        "estimated_cost": 1.8,
        "available_gpu": 1
      },
      {
        "topology_node_id": "compute-1",
        "score": 0.78,
        "baseline": {"effective_gflops": 5200, "frame_latency_p90_ms": 14.6},
        "estimated_path_latency_ms": 18.9,
        "estimated_cost": 1.2,
        "available_gpu": 1
      }
    ]
  }
}
```

UI 文案要避免说“最好节点”这种绝对判断，建议说“本策略下推荐节点”或“本次路由选择依据”。因为不同策略下好坏标准不同：成本优先不一定选最快节点，低时延转发也不一定选计算 baseline 最高的节点。

## 5. 前后端解析改造

### 5.1 前端改造

用户端对话页面需要支持：

1. 展示可用输入提示：源端和目的端可以填写节点别名或数据面 IP。
2. 目的端端口不作为主流程输入项，按任务类型固定生成；矩阵任务 `9000`，视频任务 `9100`。
3. 输入后调用后端解析接口，展示解析结果：
   - 拓扑节点 ID
   - 节点别名
   - 数据面 IP
   - 目的端固定接收端口
   - callback URL
4. 解析失败时禁止提交，提示“该业务面 IP 未登记在当前拓扑中，不能作为可路由端点”。
5. 工单详情展示 A -> B -> C：
   - A：源端拓扑节点 ID + 数据面 IP
   - B：compute 节点 + GPU + 业务面 IP/端口
   - C：目的端拓扑节点 ID + 数据面 IP + 接收端口
6. 展示目的端回调状态：
   - 未回调
   - 回调成功
   - 回调失败但平台结果已留存
7. 高级设置中支持“仅生成路由方案，不自动部署”选项。
8. 工单详情展示“路由决策说明”卡片，包含选中节点、GPU、baseline、策略匹配解释和候选节点对比。

### 5.2 后端改造

后端需要补充：

1. 端点解析能力：
   - 根据别名查数据面 IP。
   - 根据数据面 IP 反查别名和拓扑节点 ID。
   - 未登记节点/IP 直接返回校验失败。
2. DAG 生成能力：
   - source/sink 同时写入 `topology_node_id` 和 `business_ip`。
   - sink 写入 `business_port` 和 `callback_url`。
   - 用户演示模式下 source/sink 写入 `deployable=false`。
3. compute 环境变量注入：
   - 注入 callback URL 和用户端点数据面信息。
4. 回调记录：
   - 保存回调请求时间、状态码、错误信息。
   - 回调失败不覆盖平台侧业务结果。
5. 多实例区分：
   - 所有 callback body 必须携带 `order_id`。
   - receiver 和平台均按 `order_id` 区分任务。
6. 只路由不部署能力：
   - 支持 `platform_deployment.mode=route_only` 和 `deployable_roles=[]`。
   - 接收路由结果后保存 placement 和 metadata，但不物化实例。
   - 工单不进入业务目标成功率统计。
7. 路由决策展示数据聚合：
   - 从 placement、routing_result.metadata、nodes、node_baselines 聚合“路由决策说明”。
   - 当路由 metadata 缺失时，至少展示平台可查询的节点资源和 baseline。

## 6. 数据集改造

意图解析数据集需要增加用户演示模式样本，覆盖别名、数据面 IP、固定端口和回调表达。

### 6.1 样本覆盖维度

| 维度 | 要求 |
|------|------|
| 业务类型 | 矩阵计算、视频 AI 推理 |
| 源端 | 节点别名和数据面 IP 两种写法 |
| 目的端 | 节点别名、数据面 IP |
| 端口 | 按任务类型固定，矩阵 `9000`，视频 `9100` |
| 路由策略 | 成本开销保障、低时延转发、资源预留保障、完成时间优先、资源负载均衡 |
| 业务参数 | 明确参数、默认参数、口语化表达 |
| 回调表达 | “结果发到”“目的端接收”“送到目的端”等自然语言 |

### 6.2 样本示例

```text
我要做矩阵计算，源端是 h1，目的端是 h3，矩阵规模 2048，低时延优先。
```

```text
从 10.112.126.124 发起视频 AI 推理，结果发送到 10.112.20.40，使用默认测试视频。
```

```text
源端 h2，目的端 h12，做矩阵计算，结果送到目的端，资源预留保障。
```

```text
帮我跑一个视频检测任务，源端数据面 IP 是 10.112.253.42，目的端接收地址是 10.112.83.255，要求低时延转发。
```

### 6.3 解析策略

建议采用“规则优先 + 大模型补充”的方式：

- IP 和节点别名用规则提取，目的端口由任务类型固定生成。
- 节点别名用拓扑表解析。
- 业务类型、策略、业务参数由大模型或规则共同识别。
- 如果端点解析失败，即使大模型给出业务类型，也不允许提交工单。

这样可以降低大模型对 IP/端口格式的幻觉风险。

## 7. 在线 API 意图测评要求

当前计划废弃批量推理 API，改用在线 API 逐条测评。正式认为可用前，必须完成完整 360 样本验证。

### 7.1 验证目标

在线 API 测评必须证明：

- 360 条样本能够完整跑完。
- 不因限流导致任务中断。
- 支持失败重试和断点续跑。
- 输出完整测评报告。
- 页面能展示实时进度。

### 7.2 运行要求

建议默认参数：

| 参数 | 建议 |
|------|------|
| 并发数 | 正式验收按 `docs/testing.md` 使用 `8`；平台限流异常排查时可临时降到 `1` 或 `2` |
| 重试次数 | 正式验收建议 `5` |
| 重试退避 | 固定间隔或指数退避，当前脚本使用 5 秒间隔并支持断点续跑 |
| 进度记录 | 每条样本完成后落盘或写库 |
| 断点续跑 | 已完成样本不重复调用 |
| 报告 | 保存准确率、失败原因、原始响应摘要 |

不能只用 5 条或 10 条 smoke 证明“不限流”。只有完整 360 条跑通后，才能在文档和演示里说在线测评链路稳定可用。

### 7.3 前端展示

意图测评页面应展示：

- 当前运行状态
- 已完成数量 / 总数量
- 当前准确率
- 失败样本数量
- 限流/重试次数
- 最近错误信息
- 下载报告入口

页面只展示“意图参数解析准确率”，不要同时展示“大模型准确率”和“规则兜底准确率”，避免专家困惑。
